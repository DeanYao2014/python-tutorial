# 1.8 模块系统详解

## `import` 的本质：三步走

`import` 不是"把代码复制过来"，而是一个精巧的流水线——**查找器（Finder）** 找到模块文件，**加载器（Loader）** 编译并执行它，最终返回一个**模块对象（Module Object）**。

```python
import json  # 三件事在背后发生：
# 1. Finder 在 sys.path 中找到 json.py
# 2. Loader 编译 json.py → 执行 → 创建 module 对象
# 3. 把 module 对象绑定到当前作用域的名字 "json"
```

> 三步走模型由 **PEP 302** 定义，Python 3.3 后由 `importlib` 统一实现。你可以通过 `sys.meta_path` 和 `sys.path_hooks` 插入自己的 Finder/Loader——这是 Flask 自动重载、Jupyter 从数据库加载代码等机制的基础。

## `sys.modules` 缓存：为什么重复 import 不会重复执行

```python-run
import sys

len_before = len(sys.modules)
import json
print(f"首次 import json 后 sys.modules 条目增加了: {len(sys.modules) - len_before}")

import json  # 第二次——什么也不会发生
print(sys.modules["json"] is json)  # True，完全是同一个对象!
```

`sys.modules` 是一个**字典缓存**。每次 `import` 先去这里面查——命中就直接返回，不重新执行模块代码。

> 这意味着：**一个模块的顶层代码只执行一次**（在解释器的整个生命周期中）。想把模块当成"脚本"反复运行？你需要 `importlib.reload()`。

```python-run
import sys, os, math
print(f"os 在缓存里吗? {'os' in sys.modules}")        # True
print(f"os 对象的类型: {type(os)}")                   # <class 'module'>
print(f"os 有哪些属性（前 5 个）: {list(os.__dict__.keys())[:5]}")
print(f"math.__file__: {math.__file__}")              # 源码存在的路径
```

## 模块也是对象

`import` 返回的模块是一个标准的 Python 对象，有类型、有 `__dict__`、有属性：

```python-run
import math

print(type(math))                          # <class 'module'>
print(isinstance(math, object))            # True
print(math.__dict__.get("pi"))             # 3.14159... —— 所有"顶层名字"都在这
```

`module.__dict__` 就是**模块的全局命名空间**——你在模块顶层定义的变量、函数、类，本质上都在这个字典里。`module.name` 等价于 `module.__dict__["name"]`。

| 关键属性 | 含义 |
|----------|------|
| `__name__` | 模块名字符串（如 `"json"`） |
| `__file__` | 模块文件的磁盘路径 |
| `__package__` | 所属包名（顶层模块为空字符串） |
| `__path__` | 包的搜索路径列表（仅包有，普通模块没有） |
| `__spec__` | 模块规格对象（ModuleSpec），记录导入时用的 Finder/Loader |

## 绝对导入 vs 相对导入

**绝对导入（Absolute Import）**：从项目根路径出发，清楚明了——推荐首选。

```python
# myproject/utils/helper.py
from myproject.core.models import User    # 绝对导入，路径一清二楚
from collections import defaultdict       # 标准库也用绝对导入
```

**相对导入（Relative Import）**：用 `.` 和 `..` 表达层级关系，只能在**包内部**使用。

```python
# myproject/utils/helper.py
from . import config           # 同级（当前 utils/）
from ..core.models import User # 上一级（myproject/）
from ..main import app          # 上两级的同级模块
```

| 对比维度 | 绝对导入 | 相对导入 |
|----------|---------|---------|
| 可读性 | 一眼能看出模块在哪 | 需要心算层级关系 |
| 重构安全性 | 改了目录名，import 全得改 | 包内部移动文件，导入自动跟随 |
| 适用场景 | 跨包引用、公共 API | 包内部紧密耦合的模块间 |
| 能否在脚本中使用 | **可以** | **不行**——直接 `python helper.py` 会报 `ImportError` |

> 相对导入只能在包内使用，因为解释器需要 `__package__` 来确定当前层级。直接运行 `python some_module.py` 时 `__package__` 是 `None`，相对导入就崩溃了。

## `__init__.py` 的双重身份

在 Python 3.3 之前，一个目录必须包含 `__init__.py` 才是**包（Regular Package）**。3.3+ 引入 **PEP 420 命名空间包（Namespace Package）**——`__init__.py` 变成可选。

```python
# __init__.py 的三项核心职责：

# 1. 控制 from package import * 导出的名字
__all__ = ["User", "Order", "db"]

# 2. 在包加载时执行初始化（连接数据库、注册插件）
from .config import load_config
config = load_config()

# 3. 扁平化子模块接口（让用户少打一层路径）
from .core.models import User   # 这样用户就能 from mypkg import User
```

| | 常规包（有 `__init__.py`） | 命名空间包（无 `__init__.py`，PEP 420） |
|--|---------------------------|--------------------------------------|
| 加载时机 | `import` 时执行 `__init__.py` | 首次访问时才真正加载 |
| `__path__` | `['/path/to/pkg']` | 可能是多个路径（`['/path1/pkg', '/path2/pkg']`），允许跨目录拼接 |
| 适用场景 | 99% 的项目 | 大型项目跨仓库共享命名空间（如 `awscli` 和 `botocore` 共享 `aws` 前缀） |

## `__import__` vs `importlib.import_module()`

```python
# __import__ 是内置函数——它是 import 语句的底层实现
json_mod = __import__("json")        # 返回顶层包，不是 json 本身

# importlib.import_module() 是推荐的前端 API
import importlib
json_mod = importlib.import_module("json")  # 直接返回你想要的模块
```

```python-run
import importlib

# 动态导入：模块名来自字符串变量
module_name = "datetime"
m = importlib.import_module(module_name)
print(f"今天是 {m.date.today()}")

# 重新加载模块（强制重新执行顶层代码）
import calendar
cal_id_before = id(calendar)
importlib.reload(calendar)
print(f"reload 返回的是同一个对象吗? {id(calendar) == cal_id_before}")  # True
```

> **规则**：永远用 `importlib.import_module()`，不要直接调 `__import__()`。后者返回的层级在不同情形下不一致。

## `reload()` 的局限

`importlib.reload()` 重新执行模块的顶层代码，但**不会更新旧对象的引用**：

```python
# 假设 module_a.py 内容：GREETING = "hello"

from module_a import GREETING  # name 是"hello"的引用
# 改了 module_a.py 的内容后...
importlib.reload(module_a)     # module_a.GREETING 更新了
print(GREETING)                # 还是 "hello"！——旧引用没有被刷新
```

> `reload()` 只是"重跑一遍模块代码"，不是"撤销它做过的一切"。旧的 `from X import Y` 引用不会自动更新——这是 `from import` 比 `import module` 不灵活的地方之一。

## 常见的 import 陷阱

### 1. 循环导入（Circular Import）

```python
# a.py
from b import func_b      # ❌ 此时 b 还没执行完，func_b 不存在！
def func_a():
    pass

# b.py
from a import func_a      # ❌ 同理
def func_b():
    pass
```

**三种修复方案**：

```python
# 方案 1：延迟导入（把 import 移到函数内部）
def func_a():
    from b import func_b  # 调用时才导入，此时两个模块都已加载完毕
    func_b()

# 方案 2：换成 import module 而非 from module import name
import b  # import b 只是让 b 出现在命名空间，不会立刻访问 b.func_b
def func_a():
    b.func_b()  # 此时 b 已经加载完了

# 方案 3：重构——提取共同依赖到第三个模块
# 如果 a 和 b 互相需要对方，说明它们耦合太紧，应该抽出一个 c
```

### 2. `if __name__ == "__main__"` 的机制

```python-run
print(f"当前模块的 __name__: {__name__}")

if __name__ == "__main__":
    print("我以脚本身份运行（python xxx.py）")
else:
    print(f"我被 import 了（import {__name__}）")
```

当执行 `python foo.py` 时，解释器把 `foo` 这个模块的 `__name__` 设成 `"__main__"`（而不是正常的模块名）。这个惯用法让我们能在同一个文件里同时写**可运行的 demo** 和**可被 import 的代码**。

### 3. `sys.path` 的查找顺序

```python-run
import sys
for i, p in enumerate(sys.path[:6]):
    print(f"[{i}] {p}")
print(f"...（共 {len(sys.path)} 个路径）")
```

搜索顺序：
1. **当前脚本所在目录**（或交互式解释器的当前工作目录）
2. `PYTHONPATH` 环境变量中的路径
3. 标准库路径
4. `site-packages`（第三方包安装目录）

> 如果你在一个目录下写了 `test/string.py`，`import string` 会优先找到你的 `string.py` 而不是标准库的。**这是 Python 最常见的"诡异的 import 错误"根源。**

## `__pycache__` 和 `.pyc` 字节码缓存

每次 import 一个 `.py` 文件，CPython 会把它编译成**字节码（bytecode）**，缓存在 `__pycache__/` 目录下：

```
mypackage/
├── __init__.py
├── core.py
└── __pycache__/
    ├── __init__.cpython-312.pyc
    └── core.cpython-312.pyc
```

文件名编码了 Python 版本（`cpython-312`），所以**多版本共存不会互相覆盖**。

```python-run
import importlib.util
# 查看字节缓存路径（不一定已存在）
spec = importlib.util.find_spec("json")
print(f"源码路径: {spec.origin}")
print(f"缓存路径: {spec.cached}")
```

> 字节码缓存**只是性能优化**——下次 import 时如果 `.pyc` 比 `.py` 新，就跳过编译。用 `python -B` 可以禁止写 `.pyc`，用 `PYTHONDONTWRITEBYTECODE=1` 环境变量效果相同。

## 总结

`import` 系统是 Python 最被低估的子系统之一。三个关键认知：

1. **import 不是预处理指令**——它是运行时操作，可以出现在任何地方（函数内、条件分支内），返回值是一个 `module` 对象。
2. **`sys.modules` 是单例工厂**——每个模块在整个解释器生命周期中只有一个实例。
3. **循环导入的根因是模块顶层代码的执行顺序**——理解 `sys.modules` 的缓存时机，就理解了所有循环导入问题的解法。

## 本章小结

- `import` 三步走：Finder 定位模块 → Loader 编译执行 → 返回 module 对象；`sys.modules` 是全局缓存，保证每个模块只执行一次
- 绝对导入从项目根出发（推荐首选），相对导入用 `.` 表达包内层级（只能在包内使用，不能直接在脚本中运行）
- 循环导入的根本原因是模块顶层代码执行顺序——三种修复：延迟导入、改用 `import module`、提取共同依赖
- `reload()` 重新执行模块顶层代码但不会更新旧的 `from import` 引用；交互式调试时值得注意
- `python -m pkg.module` 把项目根加入 `sys.path[0]`，直接 `python pkg/module.py` 则加入当前目录——导入行为不同
