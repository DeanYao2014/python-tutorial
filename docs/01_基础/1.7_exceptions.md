# 1.7 异常体系

## 异常即对象：每个异常都是一个实例

在 Python 中，**异常（Exception）是对象**——不是错误码，不是字符串，而是一个携带了类型、消息、调用栈的完整对象。

```python-run
try:
    1 / 0
except ZeroDivisionError as e:
    print(f"类型:   {type(e)}")
    print(f"消息:   {e}")
    print(f"参数:   {e.args}")
    print(f"是否有 __traceback__: {e.__traceback__ is not None}")
```

> **一句话**：`e` 是一个 `ZeroDivisionError` 实例，它不只是一个"错误消息"——身上还挂着 `__traceback__`（记录调用栈）和 `__cause__`/`__context__`（记录异常链）。

### 内置异常的继承树

Python 的所有内置异常组织成一棵继承树，根是 `BaseException`：

```
BaseException
├── SystemExit           # sys.exit() 抛出
├── KeyboardInterrupt    # Ctrl+C
├── GeneratorExit        # 生成器关闭
└── Exception            # ← 所有常规异常的基类
    ├── ArithmeticError
    │   ├── ZeroDivisionError
    │   └── OverflowError
    ├── LookupError
    │   ├── IndexError
    │   └── KeyError
    ├── TypeError
    ├── ValueError
    ├── RuntimeError
    ├── OSError
    │   ├── FileNotFoundError
    │   └── PermissionError
    ├── ImportError
    │   └── ModuleNotFoundError  # 3.6+
    ├── StopIteration
    ├── AssertionError
    ├── AttributeError
    └── ...（还有更多）
```

```python-run
# 验证继承关系
print(f"ZeroDivisionError 是 Exception 的子类: {issubclass(ZeroDivisionError, Exception)}")   # True
print(f"ZeroDivisionError 是 BaseException 的子类: {issubclass(ZeroDivisionError, BaseException)}")  # True
print(f"KeyboardInterrupt 是 Exception 的子类: {issubclass(KeyboardInterrupt, Exception)}")  # False!
```

> `KeyboardInterrupt` 和 `SystemExit` **不是** `Exception` 的子类——这意味着 `except Exception` 抓不到它们。这是刻意的设计：Ctrl+C 和 `sys.exit()` 应该"穿透"一般异常处理。

## `try/except/else/finally` 完整语法

```python-run
def demo(order):
    results = []
    try:
        results.append("1. try 块")
        x = 10 / order  # order=0 则触发 ZeroDivisionError
        results.append("2. try 块完成")
    except ZeroDivisionError:
        results.append("3. except ZeroDivisionError")
    except Exception as e:
        results.append(f"3. except 其他: {type(e).__name__}")
    else:
        results.append("4. else — 没有异常才执行")
    finally:
        results.append("5. finally — 无论如何都执行")
    return results, x if 'x' in dir() else None

print("=== order=2 (正常) ===")
results, val = demo(2)
print("\n".join(results), f"\n返回值: {val}")

print("\n=== order=0 (异常) ===")
results, val = demo(0)
print("\n".join(results), f"\n返回值: {val}")
```

执行顺序总结：

| 场景 | 执行顺序 |
|------|---------|
| 无异常 | `try` → `else` → `finally` |
| 匹配的异常 | `try`（异常前） → `except` → `finally` |
| 不匹配的异常 | `try`（异常前） → `finally` → 异常向上传播 |
| try 中有 `return` | `finally` **仍然执行**，且在 `return` 之前 |

> `finally` 块里如果有 `return`，会"吞掉"所有未处理的异常和之前的 `return` 值——这几乎总是一个 bug。

## `__traceback__`：异常的"旅行记录"

每个异常对象的 `__traceback__` 属性记录了这个异常**从抛出点到捕获点的完整调用栈**：

```python-run
import traceback

def c():
    return 1 / 0

def b():
    return c()

def a():
    return b()

try:
    a()
except ZeroDivisionError as e:
    # __traceback__ 是一个 traceback 对象链
    tb = e.__traceback__
    frames = traceback.extract_tb(tb)
    for i, frame in enumerate(frames):
        print(f"[{i}] 文件: {frame.filename.split('/')[-1]}, "
              f"行: {frame.lineno}, 函数: {frame.name}, 代码: {frame.line}")
```

`traceback` 模块常用函数：

```python-run
import traceback

try:
    eval("int('abc')")
except ValueError as e:
    # 获取格式化字符串（不打印）
    formatted = traceback.format_exc()
    print("=== format_exc() ===")
    print(formatted)
    
    # 只获取异常类型和消息
    print("=== format_exception_only ===")
    print("".join(traceback.format_exception_only(e)))
```

> `format_exc()` 常用于日志记录：把异常信息写入日志文件，而不是直接 dump 到终端。

## 异常链：`raise ... from`

Python 的异常链分两种：**隐式链**和**显式链**。

### 隐式链（implicit chaining）

在 `except` 块中再 `raise` 一个新异常，Python 自动将原异常设为新异常的 `__context__`：

```python-run
try:
    int("abc")
except ValueError as e:
    try:
        1 / 0
    except ZeroDivisionError as e2:
        print(f"e2.__context__ is e: {e2.__context__ is e}")  # True
        print(f"隐式链: {e2.__context__}")
```

### 显式链（explicit chaining）：`raise ... from`

用 `raise NewError(...) from original` 指定 `__cause__`，明确声明"这个异常是那个异常导致的"：

```python-run
def load_config(path):
    try:
        with open(path) as f:
            return f.read()
    except FileNotFoundError as e:
        raise ValueError(f"配置文件 {path} 不存在") from e

try:
    load_config("/不存在的路径/config.yaml")
except ValueError as e:
    # 显式链：__cause__ 指向原异常
    print(f"异常: {e}")
    print(f"__cause__: {e.__cause__}")
    print()
    # Python 默认打印时会展示完整异常链
    import traceback
    traceback.print_exc()
```

> `raise ... from e` 的语义是：**"翻译"** 异常——把底层实现细节转换成上层能理解的业务异常。

### 切断异常链：`raise ... from None`

当你不想暴露内部异常细节时，用 `from None` 显式切断：

```python-run
def validate(x):
    try:
        int(x)
    except ValueError:
        raise TypeError(f"期望整数，收到 {x!r}") from None

try:
    validate("abc")
except TypeError as e:
    print(f"异常: {e}")
    print(f"__cause__: {e.__cause__}")    # None
    print(f"__context__: {e.__context__}")  # None——链被切断
```

## Python 3.11+ ExceptionGroup + `except*`

传统 `try/except` 一次只能处理一个异常。如果同时并发执行多个任务，可能**同时发生多个不相关的异常**。Python 3.11 引入 `ExceptionGroup` 和 `except*` 来解决这个问题：

```python-run
# 注意：此代码需要 Python 3.11+

# 创建包含多个异常的分组
eg = ExceptionGroup(
    "多个任务失败",
    [ValueError("值错误 A"), TypeError("类型错误 B"), OSError("系统错误 C")]
)

try:
    raise eg
except* ValueError as e:
    print(f"[ValueError 分支] 捕获: {e.exceptions}")
except* TypeError as e:
    print(f"[TypeError 分支]  捕获: {e.exceptions}")
except* Exception as e:
    print(f"[兜底分支]        捕获: {e.exceptions}")
```

> `except*` 不是"匹配第一个异常就结束"，而是**遍历 ExceptionGroup 中的所有异常**，每个 `except*` 捕获匹配的那部分。一个 ExceptionGroup 可以被多个 `except*` 分头处理。

### 真实场景：并发任务

```python-run
# 模拟多个并发任务，其中一部分失败
def task(name, fail=False):
    if fail:
        raise RuntimeError(f"{name} 失败了")
    return f"{name} 完成"

errors = []
results = []
for name, fail in [("任务A", False), ("任务B", True), ("任务C", True)]:
    try:
        results.append(task(name, fail))
    except RuntimeError as e:
        errors.append(e)

if errors:
    try:
        raise ExceptionGroup("批次任务中有失败", errors)
    except* RuntimeError as eg:
        print(f"总共 {len(eg.exceptions)} 个任务失败:")
        for err in eg.exceptions:
            print(f"  - {err}")
```

## Python 3.11+ `add_note()`：给异常附加上下文信息

当捕获异常后想补充业务相关信息（但不想改变异常类型），用 `add_note()`：

```python-run
# 需要 Python 3.11+

def process_user(uid):
    users = {"alice": 28, "bob": 35}
    try:
        return users[uid]
    except KeyError as e:
        e.add_note(f"查询的用户ID: {uid!r}")
        e.add_note(f"当前已注册用户: {list(users)}")
        raise

try:
    process_user("charlie")
except KeyError as e:
    import traceback
    traceback.print_exc()
```

输出中的每个 note 单独一行，清晰地挂在异常消息之后。相比"在 raise 时拼一大段字符串"，`add_note()` 的好处：

1. **非破坏性**：不修改原始异常消息，notes 追加在 `__notes__` 属性中
2. **分层添加**：不同调用层各加各的上下文，而不是一个地方拼完所有信息
3. **Python 自动打印**：`traceback.print_exc()` 会自动展示所有 notes

## 自定义异常的最佳实践

```python-run
class ConfigError(Exception):
    """配置文件相关异常的基类。"""
    pass

class ConfigNotFoundError(ConfigError):
    """配置文件不存在。"""
    def __init__(self, path, searched_dirs=None):
        self.path = path
        self.searched_dirs = searched_dirs or []
        super().__init__(self._format())
    
    def _format(self):
        msg = f"配置文件未找到: {self.path}"
        if self.searched_dirs:
            msg += f"\n已搜索的目录:\n" + "\n".join(f"  - {d}" for d in self.searched_dirs)
        return msg

# 使用时：
try:
    raise ConfigNotFoundError("db.yaml", ["/etc/app", "./config"])
except ConfigNotFoundError as e:
    print(e)
    print(f"访问 path 属性: {e.path}")
```

**关键原则：**

1. **继承 `Exception`**，不是 `BaseException`——后者包括 `SystemExit` 和 `KeyboardInterrupt`
2. **异常层级体现业务语义**：先定义基类（如 `ConfigError`），再派生子类——让调用者可以按粒度捕获
3. **`__str__` 要信息丰富**：方便日志记录和调试
4. **把原始数据挂在属性上**（如 `self.path`）：调用者可能需要编程式访问，而不是解析字符串

## 常见反模式

### 1. bare `except:` —— 吞掉 Ctrl+C

```python
# ❌ 危险：连 KeyboardInterrupt 和 SystemExit 都抓
try:
    do_something()
except:        # 等价于 except BaseException
    pass

# ✅ 正确：只抓常规异常
try:
    do_something()
except Exception as e:
    logger.error("出错: %s", e)
```

### 2. `except Exception: pass` —— 沉默的错误黑洞

```python
# ❌ 最臭名昭著的反模式
try:
    critical_operation()
except Exception:
    pass  # 出了事也没人知道
```

如果确实要忽略异常，至少**记录日志**并**只抓特定类型**：

```python
# ✅ 选择性忽略
import logging
try:
    cleanup_temp_file()
except FileNotFoundError:
    logging.debug("临时文件已被删除，忽略")
```

### 3. `raise e` —— 破坏 traceback

```python
try:
    risky()
except SomeError as e:
    # ❌ raise e → traceback 从这一行重新开始，丢失原始调用栈
    raise e

    # ✅ raise → 保留原始 traceback
    raise

    # ✅ raise NewError(...) from e → 显式链，两者都保留
    raise WrappedError("包装后的错误") from e
```

**对比效果：**

| 写法 | 原始 traceback | 语义 |
|------|:---:|---|
| `raise` | 保留 | 原样重新抛出 |
| `raise e` | **丢失**，从这一行重新开始 | 几乎总是 bug |
| `raise X(...) from e` | 保留在 `__cause__` 中 | 显式包装/翻译 |

## EAFP：Python 的错误处理哲学

Python 社区偏好 **EAFP（Easier to Ask Forgiveness than Permission）**——"先斩后奏"：

```python-run
# ❌ LBYL（Look Before You Leap）——"三思而后行"
def get_item_lbyl(d, key):
    if key in d:                    # 第一次查询字典
        return d[key]               # 第二次查询字典
    return None

# ✅ EAFP（Easier to Ask Forgiveness than Permission）——"先斩后奏"
def get_item_eafp(d, key):
    try:
        return d[key]               # 一次查询，正常路径
    except KeyError:
        return None                 # 异常路径
```

> EAFP 不是"偷懒"，背后有工程上的理由：1) 避免竞态条件（检查和使用之间状态可能改变），2) 正常路径更高效（异常不触发时几乎零开销），3) 代码主逻辑清晰，异常处理在 `except` 中安静等待。

---

> **总结**：异常是 Python 错误处理的一等公民——不是"额外的错误处理机制"，而是**唯一的错误处理机制**。掌握 `raise ... from` 构建清晰的异常链、用 ExceptionGroup 处理并发错误、用 `add_note()` 丰富上下文，写出"异常时也能优雅排查问题"的代码。

## 本章小结

- 异常是对象——不只是错误消息，还携带 `__traceback__`（调用栈）、`__cause__`/`__context__`（异常链）
- `except Exception` 抓不到 `KeyboardInterrupt` 和 `SystemExit`——它们是 `BaseException` 的直接子类，这是有意设计
- `raise ... from e` 显式翻译异常（保留原始 `__cause__`）；`raise ... from None` 切断异常链（隐藏内部细节）
- Python 3.11+ 的 `ExceptionGroup` + `except*` 一次处理多个并发异常；`add_note()` 给异常附加业务上下文
- EAFP（先斩后奏）优于 LBYL（三思后行）：避免竞态条件，正常路径零开销，主逻辑不被检查代码淹没
