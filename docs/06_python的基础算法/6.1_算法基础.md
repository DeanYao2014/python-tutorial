## 1.1 Python 内置数据结构的复杂度速查

**选错数据结构是 Python 性能问题最常见的根源**。下面这张表，值得刻在大脑里：

| 操作 | `list` | `tuple` | `dict` | `set` | `deque` |
|------|--------|---------|--------|-------|---------|
| `x in container` | O(n) | O(n) | **O(1)** | **O(1)** | O(n) |
| `container[i]` | **O(1)** | **O(1)** | — | — | O(1) |
| 末尾追加 | **O(1)** | — | — | — | **O(1)** |
| 头部插入 | O(n) | — | — | — | **O(1)** |
| 中间插入/删除 | O(n) | — | — | — | O(n) |
| 遍历 | O(n) | O(n) | O(n) | O(n) | O(n) |
| 获取长度 `len()` | **O(1)** | **O(1)** | **O(1)** | **O(1)** | **O(1)** |

```python-run
from collections import deque
import time

n = 100_000
lst = list(range(n))
dq = deque(range(n))

# 头部删除：list O(n) vs deque O(1)
t0 = time.perf_counter()
for _ in range(1000):
    lst.pop(0)          # 每次都要左移所有元素
t1 = time.perf_counter()

t2 = time.perf_counter()
for _ in range(1000):
    dq.popleft()        # 只改指针，常数时间
t3 = time.perf_counter()

print(f"list.pop(0) 1000次: {t1 - t0:.4f}s")
print(f"deque.popleft() 1000次: {t3 - t2:.6f}s")
```

**结论**：如果你需要频繁在头部插入/删除，`list` 是错误选择——`deque` 才是正确答案。

## 1.2 `timeit`：Python 的秒表

`timeit` 模块是 Python 标准的性能测量工具。它自动运行代码多次、关闭 GC、在重复循环中测量，避免了"只测一次"的各种陷阱。

```python-run
import timeit

# timeit 接受字符串形式的代码和 setup
# setup的核心作用是“把不计入计时的初始化代码，和需要计时的代码剥离开”
setup = "data = list(range(10_000))"

# 成员检查：list O(n) vs set O(1)
t_list = timeit.timeit("9999 in data", setup, number=1000)
t_set  = timeit.timeit("9999 in set(data)", setup, number=1000)
# 推荐使用 global + lambda
data = list(range(10_000))
def demo(val):
    return val in data

t_list = timeit.timeit(lambda : 9999 in data, globals=globals(),number=1000 )
# 参数可以通过默认值的方式给函数传递
t_list2 = timeit.timeit(lambda val=9999: demo(val), globals=globals(),number=1000 )

print(f"list 成员检查 1000次: {t_list:.6f}s")
print(f"set  成员检查 1000次: {t_set:.6f}s")
print(f"差距: {t_list / t_set:.0f} 倍")
print(f"set  成员检查 1000次: {t_list:.6f}s")
print(f"set  成员检查 1000次: {t_list2:.6f}s")

```

**关键原则**：
- `timeit` 用于测量**微秒级的代码片段**
- `cProfile` 用于分析**整个程序的瓶颈在哪里**
- 先测再优化——猜的性能瓶颈 80% 是错误的

> 📖 **官方文档**：[timeit — Measure execution time of small code snippets](https://docs.python.org/3/library/timeit.html)

## 1.3 常数因子不可忽视

大 O 忽略常数因子，但 Python 中的常数因子经常大到改变实际选择：

```python-run
import time

n = 10_000_000

# 内联加法
t0 = time.perf_counter()
total = 0
for i in range(n):
    total += i
t1 = time.perf_counter()

# 函数调用加法
def add(a, b):
    return a + b

t2 = time.perf_counter()
total = 0
for i in range(n):
    total = add(total, i)
t3 = time.perf_counter()

print(f"内联操作: {t1 - t0:.3f}s")
print(f"函数调用: {t3 - t2:.3f}s")
```

同样是 O(n)，**Python 函数调用开销使速度差了 2-3 倍**。这就是为什么内建函数（用 C 写的）比 Python 手写循环快得多——`sum()` 比 `for` + `+=` 快一个数量级。

**实战启示**：
- 能用内置函数就用内置函数（C 实现，无解释器开销）
- 热点代码中避免循环体中的 Python 函数调用
- 列表推导式比 `for` + `append` 快（因为推导式的迭代过程在 C 层面执行）

## 1.4 `@lru_cache`：一行代码从 O(2^n) 到 O(n)

记忆化（Memoization）是算法优化的经典技巧。Python 标准库提供了开箱即用的方案：

```python-run
from functools import lru_cache
import time

# ❌ 无缓存：O(2^n)，n=35 就要几秒
def fib_slow(n):
    if n <= 1:
        return n
    return fib_slow(n - 1) + fib_slow(n - 2)

# ✅ 加一行装饰器：O(n)，n=1000 也瞬间完成
@lru_cache(maxsize=None)
def fib_fast(n):
    if n <= 1:
        return n
    return fib_fast(n - 1) + fib_fast(n - 2)

# 对比 n=30
t0 = time.perf_counter()
print(f"fib_slow(30) = {fib_slow(30)}")
t1 = time.perf_counter()
print(f"  耗时: {t1 - t0:.4f}s")

t0 = time.perf_counter()
print(f"fib_fast(30) = {fib_fast(30)}")
t1 = time.perf_counter()
print(f"  耗时: {t1 - t0:.6f}s")

# 查看缓存命中情况
print(f"\n缓存统计: {fib_fast.cache_info()}")
```

`lru_cache` 用一个字典存储"参数 → 返回值"的映射。递归前先查字典，命中则直接返回。这是**动态规划自顶向下**的 Python 最佳实践——比你手动写 DP 表更简洁且更不易出错。

> `lru_cache (least recently used cache)` 
> + 会将所有位置参数和关键字参数(参数必须是不可变参数)打包成一个元素,然后对这个元素hash运算生成最终的缓存键
> + 纯函数无副作用,输入相同结果必须相同

 
> 📖 **官方文档**：[functools.lru_cache](https://docs.python.org/3/library/functools.html#functools.lru_cache)

## 1.5 性能分析工具链

好的性能分析不只是"测一下时间"。Python 提供了分层工具：

```
timeit          → 微秒级精确测量小段代码
cProfile        → 函数级调用统计，找到哪个函数最耗时
line_profiler   → 逐行分析，精确到每条语句
tracemalloc     → 追踪内存分配，分析内存使用
```

`cProfile` : 能精确记录每个函数的调用次数和耗时，用真实数据帮你揪出“性能刺客”
```python
# cProfile 快速使用（命令行或代码内）
# python -m cProfile -s cumulative my_script.py

import cProfile
import pstats

def slow_function():
    total = 0
    for i in range(1_000_000):
        total += i ** 2
    return total

def main():
    for _ in range(10):
        slow_function()

# cProfile.run('main()', sort='cumulative')
```

## 本章小结

1. **`list` 头部操作是 O(n)**，头部频繁增删用 `deque`；**`dict`/`set` 成员检查是 O(1)**，需要频繁"是否存在"时别用 `list`
2. **`timeit` 测微秒**，`cProfile` 找瓶颈——先测再优化，别猜
3. **Python 函数调用有常数开销**：内置 C 函数 > 列表推导 > `for` + 调用 Python 函数——热点代码优先用内置
4. **`@lru_cache` 是记忆化的标准答案**：一行装饰器把 O(2^n) 递归变成 O(n)，比手动写 DP 表更简洁
5. **大 O 告诉你趋势，但 Python 的常数因子决定实际选择**：n 不够大时，O(n) 的内置 C 函数可能比 O(log n) 的手写 Python 更快

## 思考题

1. `x in lst` 是 O(n)，`x in set_of_lst` 是 O(1)。为什么有人还是用 `list` 做成员检查？什么场景下 `list` 反而更优？
> 1. 数据量极小: list的连续内存带来的cpu缓存命中率极高, 实际遍历速度甚至可能快于set hash计算和哈希冲突指针跳转带来的开销
> 2. 构建成本敏感: set的构建需要计算哈希并处理冲突 耗时O(n), 也就是构建set+查询的总成本 和遍历list的成本对比 
2. 下面两段代码都是 O(n²)，但实际速度差了 10 倍以上。为什么？
   ```python
   # A: 列表推导
   result = [x * 2 for x in range(n)]
   # B: for + append
   result = []
   for x in range(n):
       result.append(x * 2)
   ```

> 答: 列表推导是C层面优化,效率更高; ptyhon虚拟机指令解析 + append函数调用开销
3. `@lru_cache` 对非纯函数（有副作用或依赖外部状态的函数）能正常工作吗？为什么？
> 答: 能正常工作,但是会产生严重的逻辑错误; `@lru_cache`只根据 函数引用+参数(不可变)来生成哈希key,只要key没变 直接就返回结果; 如果每次调用结果不相同,那么就会缓存里旧的结果, 函数不会再次执行.
4. 你有一个 100 万元素的 `list`，你需要频繁在头部插入元素。有哪两种改善方案？
> 答: 推荐`from collections import deque` 使用双端队列: 支持pop和popleft,时间复杂度`O(1)` ; 使用reverse() + append() -> 把头部插入 变成尾部的append (O(1))的; 创建一个足够大的列表, 一个维护指针head,插入头部的时候只需要把head指针往前移动一位;
```python-run
class OffsetList:
    def __init__(self, max_capacity=100000):
        # 1. 预先分配足够大的内存空间
        self._data = [None] * max_capacity
        # 2. 维护一个头部指针, 初始指向列表的中间
        self._head=max_capacity // 2
        # 3. 记录当前的实际存储的元素个数
        self._size = 0
    
    def prepend(self,val):
        '''在头部插入元素 O(1)'''
        self._head -= 1 # 1. 指针往前移动
        self._data[self._head] = val # 2. 写入数据
        self._size += 1 # 3. 修改实际数据长度
        
    def get(self,index):
        '''按逻辑索引读取元素 O(1)'''
        if index < 0 or index >= self._size:
            raise IndexError('Index out of range')
        # 物理索引 = 逻辑索引 + 头部偏移量
        return self._data[self._head + index]
    
    def to_list(self):
        '''倒出有效数据为普通的list'''
        return self._data[self._head : self._head+self._size]

o1 = OffsetList(max_capacity=200)
o1.prepend('C')
o1.prepend('B')
o1.prepend('A')

print('逻辑读取', o1.get(0), o1.get(1), o1.get(2) )
print('倒出列表', o1.to_list())

```
