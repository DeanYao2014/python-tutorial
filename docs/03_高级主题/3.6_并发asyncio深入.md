# 3.6 并发：asyncio 深入

如果你只花时间深入学习一个并发模型，就应该是 `asyncio`。它不仅是 Python 异步编程的基础设施，更代表了一种植入 Python 语言的并发哲学：**协程（Coroutine）**。本章从协程的本质讲起，一步步深入到事件循环、Task、TaskGroup，以及 3.11-3.13 最新语法。

## Part A: 协程本质

### 协程不是线程

**线程（Thread）** 是操作系统调度的——操作系统决定哪个线程在哪个 CPU 核心上运行多久（抢占式调度）。一个线程随时可能被暂停，你控制不了。

**协程（Coroutine）** 是用户自己调度的——协程自己决定什么时候暂停（`await`），什么时候恢复。没有操作系统层面的切换开销（上下文切换不涉及内核态切换）。

| | 线程 (Thread) | 协程 (Coroutine) |
|---|---|---|
| 调度者 | 操作系统（抢占式） | 用户代码（协作式） |
| 切换开销 | ~微秒级（内核态切换） | ~纳秒级（函数调用级别） |
| 并发数上限 | 几百到几千（受栈内存限制） | 数万到数十万 |
| 适合场景 | I/O 密集 + 共享状态 | I/O 密集 + 高并发连接 |
| 共享状态 | 直接共享内存（需锁） | 同事件循环内的协程天然安全（不需要锁） |

> **核心思想**：协程是用"协作"换"效率"。每次 `await` 就是一个明确的"我愿意让出控制权"的信号。没有抢占，就没有锁竞争的开销；没有内核态切换，就能支撑海量并发。

### async def 创建了什么

```python-run
async def hello() -> str:
    await asyncio.sleep(0)
    return "hello"

coro = hello()
print(f"type: {type(coro)}")           # <class 'coroutine'>
print(f"可以 await: {hasattr(coro, '__await__')}")  # True
```

`async def` 创建的**不是**函数——是一个**协程函数（Coroutine Function）**。调用它返回的是一个**协程对象（Coroutine Object）**，而不是函数的返回值。

```python-run
# 协程对象不会自己执行——必须被 await 或事件循环驱动
import asyncio

async def greet(name: str) -> str:
    return f"Hello, {name}"

coro = greet("World")
# print(coro)  # <coroutine object greet at 0x...> —— 没执行

# 三种驱动方式：
# 1. asyncio.run() —— 创建事件循环并运行顶层协程
result = asyncio.run(coro)
print(f"asyncio.run: {result}")

# 2. await —— 在另一个协程中等待
async def wrapper():
    result = await greet("Alice")
    print(f"await: {result}")

asyncio.run(wrapper())

# 3. 手动 send() —— 仅用于理解底层，不要在生产代码中用
async def simple() -> str:
    return "done"

c = simple()
try:
    c.send(None)  # 启动协程
except StopIteration as e:
    print(f"手动驱动: {e.value}")  # StopIteration 的 value 是返回值
```

### 事件循环的本质：30 行最小实现

```python-run
from collections import deque
from typing import Any

class MiniLoop:
    """极简事件循环 —— 揭示 asyncio 的核心原理"""

    def __init__(self) -> None:
        self._ready: deque[Any] = deque()   # 就绪协程队列

    def create_task(self, coro: Any) -> None:
        self._ready.append(coro)

    def run(self) -> None:
        """不断从队列取出协程、推进一步、放回队列"""
        while self._ready:
            coro = self._ready.popleft()
            try:
                coro.send(None)           # 推进一步
                self._ready.append(coro)  # 还没完？放回去等下次
            except StopIteration:
                pass  # 协程结束了

# 使用这个极简事件循环
async def counter(name: str, n: int) -> None:
    for i in range(n):
        print(f"[{name}] {i}")
        # 主动让出 —— 在真实的 asyncio 中这就是 await asyncio.sleep(0)
        # 但在 MiniLoop 中我们模拟这个行为
        await asyncio.sleep(0)  # 这句在 MiniLoop 中不生效，仅作示意

print("事件循环本质:")
print("队列 + 轮转 + send(None) = 协程调度器")
print("真实的 asyncio 在此基础上加入了 I/O 多路复用、定时器、Future 等")
```

> **事件循环的真面目**：一个无限循环，维护一个就绪队列，不断取出协程、推进一步，处理完了放回队列尾部。真实的 `asyncio` 事件循环还整合了 `epoll`/`kqueue`/`IOCP`（I/O 多路复用）、`TimerHeap`（定时器堆）、`Future`（异步结果）——但核心逻辑就是上面这 30 行。

### 事件循环与 I/O 多路复用

真实的 `asyncio` 事件循环不只是轮转协程。它使用操作系统的 I/O 多路复用机制：

```
┌──────────────────────────────────────┐
│           事件循环主循环               │
│                                      │
│  1. 检查就绪队列 → 运行协程           │
│  2. 协程 await → 挂起，注册到 selector │
│  3. selector 等待（epoll/kqueue）     │
│  4. I/O 就绪 → 恢复协程，放回就绪队列  │
│  5. 回到步骤 1                        │
└──────────────────────────────────────┘
```

> 📖 官方文档：[Coroutines and Tasks](https://docs.python.org/3/library/asyncio-task.html), [Event Loop](https://docs.python.org/3/library/asyncio-eventloop.html)

## Part B: Task 和 Future

### create_task —— 把协程"提交"给事件循环

`await` 是**串行等待**：调用者暂停，等被调用者完成，才继续。

`create_task` 才是**并发**：把协程包装成 Task，提交给事件循环立即调度，调用者不等待。

```python-run
import asyncio, time

async def fetch(url: str) -> str:
    await asyncio.sleep(0.2)
    return f"OK({url})"

async def serial_vs_concurrent() -> None:
    # 串行 —— 每个 await 都暂停调用者
    start = time.perf_counter()
    r1 = await fetch("a.com")   # 等 a 完成...
    r2 = await fetch("b.com")   # 才轮到 b
    print(f"串行: {r1}, {r2}, 耗时 {time.perf_counter() - start:.3f}s (~0.4s)")

    # 并发 —— create_task 让两个协程同时跑
    start = time.perf_counter()
    t1 = asyncio.create_task(fetch("a.com"))
    t2 = asyncio.create_task(fetch("b.com"))
    r1 = await t1  # 等 t1 完成（同时 t2 也在跑）
    r2 = await t2  # 等 t2 完成
    print(f"并发: {r1}, {r2}, 耗时 {time.perf_counter() - start:.3f}s (~0.2s)")

asyncio.run(serial_vs_concurrent())
```

> **黄金规则**：先 `create_task`，后 `await`。如果你先 `await` 了，那并发窗口就已经关上了。

### gather —— 批量并发，收集结果

```python-run
import asyncio

async def fetch(id: int) -> str:
    await asyncio.sleep(0.1)
    return f"result-{id}"

async def demo() -> None:
    # gather 并发执行所有协程，返回结果列表（保序）
    results = await asyncio.gather(
        fetch(1),
        fetch(2),
        fetch(3),
    )
    print(f"gather: {results}")

    # gather 的默认行为：一个异常，整个 gather 抛出
    async def fail() -> str:
        raise ValueError("boom")

    try:
        await asyncio.gather(fetch(1), fail(), fetch(3))
    except ValueError as e:
        print(f"gather 碰到异常: {e}")
        # 注意：fetch(1) 和 fetch(3) 可能已经完成，也可能还在跑
        # gather 不会取消其他任务！

asyncio.run(demo())
```

### TaskGroup (3.11+) —— 结构化并发的新标准

`asyncio.TaskGroup` 解决了 `gather` 的核心缺陷：**一个任务出错时，所有兄弟任务自动取消**。

```python-run
import asyncio

async def demo() -> None:
    async def work(id: int, fail: bool = False) -> str:
        await asyncio.sleep(0.1)
        if fail:
            raise ValueError(f"任务 {id} 失败了")
        return f"result-{id}"

    # ── 正常场景 ──
    async with asyncio.TaskGroup() as tg:
        t1 = tg.create_task(work(1))
        t2 = tg.create_task(work(2))
        t3 = tg.create_task(work(3))
    # 离开 async with 时，所有任务已完成
    print(f"TG 正常: {t1.result()}, {t2.result()}, {t3.result()}")

    # ── 异常场景：TaskGroup 自动取消其他任务 ──
    results: list[str] = []
    cleanup_log: list[str] = []

    async def work_with_cleanup(id: int, fail: bool = False) -> None:
        try:
            await asyncio.sleep(0.05 * id)
            if fail:
                raise ValueError(f"任务 {id} 失败了")
            results.append(f"result-{id}")
        except asyncio.CancelledError:
            cleanup_log.append(f"任务 {id} 被取消（清理资源）")
            raise  # 重新抛出——让 TaskGroup 知道这个任务结束了

    try:
        async with asyncio.TaskGroup() as tg:
            tg.create_task(work_with_cleanup(1))
            tg.create_task(work_with_cleanup(2, fail=True))  # ← 这个会失败
            tg.create_task(work_with_cleanup(3))
    except* Exception as eg:       # except* 捕获 ExceptionGroup (3.11+)
        print(f"TG 异常: {eg}")

    print(f"完成的任务: {results}")
    print(f"清理日志: {cleanup_log}")
    # 关键：任务 1 和 3 被自动取消（如果还没完成的话）
    # CancelledError 给了它们清理资源的机会

asyncio.run(demo())
```

| | `asyncio.gather()` | `asyncio.TaskGroup()` |
|---|---|---|
| Python 版本 | 3.7+ | 3.11+ |
| 某个任务异常时 | 抛出异常，其他任务**继续运行** | 抛出 ExceptionGroup，其他任务**自动取消** |
| 异常类型 | 单个 Exception | ExceptionGroup（包含所有异常） |
| 有序性 | 结果保持输入顺序 | Task 的 `.result()` 分别获取 |
| 推荐度 | 简单场景还可以用 | 新代码的**默认选择** |

> `TaskGroup` 的设计灵感来自 Trio 的 **Nursery**——结构化并发的理念：每个协程的"生命周期"被 `async with` 的上下文严格限定，离开这个上下文时，所有子协程都必须已经结束。

> 📖 官方文档：[TaskGroup](https://docs.python.org/3/library/asyncio-task.html#task-groups), [PEP 654 – Exception Groups and except*](https://peps.python.org/pep-0654/)

### wait_for —— 给协程加超时

```python-run
import asyncio

async def slow_work() -> str:
    await asyncio.sleep(1.0)
    return "太慢了"

async def demo() -> None:
    try:
        result = await asyncio.wait_for(slow_work(), timeout=0.2)
    except asyncio.TimeoutError:
        print("超时！协程被自动取消")

    # wait_for 在超时时会自动取消内部协程
    # 内部协程会收到 CancelledError，有机会做清理

asyncio.run(demo())
```

### as_completed —— 流式处理

```python-run
import asyncio

async def fetch(id: int, delay: float) -> str:
    await asyncio.sleep(delay)
    return f"result-{id}"

async def demo() -> None:
    tasks = [
        asyncio.create_task(fetch(1, 0.3)),
        asyncio.create_task(fetch(2, 0.1)),
        asyncio.create_task(fetch(3, 0.2)),
    ]

    # as_completed: 哪个先完成，先处理哪个
    # 不需要等慢的任务
    for coro in asyncio.as_completed(tasks):
        result = await coro
        print(f"最先完成: {result}")

asyncio.run(demo())
```

## Part C: 最新推荐语法 (Python 3.11-3.13)

这一节是本章的核心——展示从旧写法到新推荐写法的演进。

### 3.11: `asyncio.to_thread()` —— 同步代码的异步桥梁

```python-run
import asyncio, time

def blocking_io(id: int) -> str:
    """同步阻塞 I/O —— 不能在事件循环中直接调用"""
    time.sleep(0.2)  # 同步 sleep！
    return f"sync-result-{id}"

async def old_way() -> None:
    """旧方式: loop.run_in_executor —— 啰嗦"""
    loop = asyncio.get_event_loop()
    result = await loop.run_in_executor(None, blocking_io, 1)
    print(f"旧方式: {result}")

async def new_way() -> None:
    """新方式: asyncio.to_thread —— 简洁、语义清晰 (3.11+)"""
    result = await asyncio.to_thread(blocking_io, 1)
    print(f"新方式: {result}")

    # 并发运行多个同步任务
    tasks = [asyncio.to_thread(blocking_io, i) for i in range(3)]
    results = await asyncio.gather(*tasks)
    print(f"并发: {results}")

asyncio.run(new_way())
```

`asyncio.to_thread()` 底层还是 `run_in_executor()`，但语义更清晰："把这个同步调用扔到线程池里去，别卡事件循环"。

### 3.11: `except*` —— 处理 ExceptionGroup

```python
# 旧世界: 一个异常，全局捕获
try:
    await asyncio.gather(task1, task2, task3)
except ValueError:
    # 但如果有两个任务都抛出了 ValueError 呢？你只能捕获到一个。
    ...

# 新世界: ExceptionGroup + except*
try:
    async with asyncio.TaskGroup() as tg:
        tg.create_task(maybe_fail("a"))
        tg.create_task(maybe_fail("b"))
except* ValueError as eg:
    for exc in eg.exceptions:
        print(f"捕获到: {exc}")
```

### 3.12: `eager_task_factory` —— 更快的任务创建

```python-run
import asyncio

async def demo() -> None:
    # 默认行为: create_task 后，任务在下一轮事件循环才执行第一步
    # eager 行为: create_task 后立即执行第一步（直到遇到 await 为止）
    loop = asyncio.get_event_loop()
    loop.set_task_factory(asyncio.eager_task_factory)

    async def quick() -> str:
        return "instant"

    task = asyncio.create_task(quick())
    result = await task
    print(f"eager: {result}")

    # 恢复默认
    loop.set_task_factory(asyncio.Task)

asyncio.run(demo())
```

适用于短平快的任务——省去一次事件循环的往返调度。

### 3.13: `asyncio.timeout()` 上下文管理器

```python-run
import asyncio

async def demo() -> None:
    # 旧方式: asyncio.wait_for() —— 包裹整个调用
    # result = await asyncio.wait_for(slow_op(), timeout=5.0)

    # 新方式: asyncio.timeout() 上下文管理器 —— 更灵活的作用域控制 (3.13+)
    try:
        async with asyncio.timeout(0.1):
            await asyncio.sleep(0.05)   # 在超时内完成
            print("步骤 1: 完成")
            await asyncio.sleep(0.05)   # 也在超时内
            print("步骤 2: 完成")
    except TimeoutError:
        print("不应该超时")

    # 真正超时
    try:
        async with asyncio.timeout(0.1):
            await asyncio.sleep(0.3)    # 超过 0.1s
    except TimeoutError:
        print("超时了！")

    # timeout() 的优势: 可以包裹多步操作，甚至可以嵌套
    # 旧 wait_for 只能包裹单个 awaitable

asyncio.run(demo())
```

> 如果运行在 Python 3.12 或更早，`asyncio.timeout()` 不可用。替代方案：`await asyncio.wait_for(slow_op(), timeout=5.0)`。

## Part D: 实战模式

### 模式一：生产者-消费者 (asyncio.Queue)

```python-run
import asyncio

async def demo() -> None:
    async def producer(q: asyncio.Queue[int], n: int) -> None:
        """生产 n 个物品"""
        for i in range(n):
            await asyncio.sleep(0.05)  # 生产耗时
            await q.put(i)
            print(f"生产: {i}")
        await q.put(None)  # 哨兵（Sentinel）: 通知消费者结束

    async def consumer(q: asyncio.Queue[int], name: str) -> None:
        """消费物品，直到收到 None"""
        while True:
            item = await q.get()
            if item is None:
                await q.put(None)  # 把哨兵放回去，让其他消费者也能看到
                break
            await asyncio.sleep(0.1)  # 消费耗时
            print(f"[{name}] 消费: {item}")

    q: asyncio.Queue[int] = asyncio.Queue(maxsize=3)  # 限制容量 = 背压控制
    async with asyncio.TaskGroup() as tg:
        tg.create_task(producer(q, 6))
        tg.create_task(consumer(q, "A"))
        tg.create_task(consumer(q, "B"))

asyncio.run(demo())
```

`maxsize=3` 是**背压（Backpressure）**：如果消费者慢，生产者会在 `q.put(i)` 处**自动等待**，直到队列有空位。这防止了"生产者跑得太快、撑爆内存"的问题。

### 模式二：速率限制 (asyncio.Semaphore)

```python-run
import asyncio

async def demo() -> None:
    semaphore = asyncio.Semaphore(3)  # 最多 3 个并发

    async def limited_request(id: int) -> str:
        async with semaphore:          # 获取名额（可能等待）
            await asyncio.sleep(0.2)   # 模拟网络请求
            return f"请求 {id} 完成"

    tasks = [asyncio.create_task(limited_request(i)) for i in range(10)]
    results = await asyncio.gather(*tasks)
    print(f"限速完成: {results}")

asyncio.run(demo())
```

应用场景：API 限流（每秒最多 N 个请求）、数据库连接池（最多 M 个连接）。

### 模式三：优雅关闭 (Graceful Shutdown)

```python-run
import asyncio, signal

async def demo() -> None:
    stop = asyncio.Event()

    async def worker(name: str) -> None:
        count = 0
        while not stop.is_set():
            await asyncio.sleep(0.1)  # 模拟工作
            count += 1
        print(f"[{name}] 处理了 {count} 轮，正在退出")

    # 2 秒后自动触发停止
    async def auto_stop() -> None:
        await asyncio.sleep(2)
        stop.set()

    async with asyncio.TaskGroup() as tg:
        tg.create_task(worker("A"))
        tg.create_task(worker("B"))
        tg.create_task(auto_stop())

    print("所有 worker 已退出")

asyncio.run(demo())
```

生产环境中，`stop.set()` 由 `SIGINT`/`SIGTERM` 的信号处理器触发。

### 模式四：混合同步/异步代码

```python-run
import asyncio, time

def legacy_sync_function(data: str) -> str:
    """遗留的同步函数——你不能改它，但它会阻塞"""
    time.sleep(0.2)
    return data.upper()

async def demo() -> None:
    # to_thread 把同步调用变成 async 协程
    result = await asyncio.to_thread(legacy_sync_function, "hello")
    print(f"混合: {result}")

    # 并发运行多个同步任务
    results = await asyncio.gather(
        asyncio.to_thread(legacy_sync_function, "a"),
        asyncio.to_thread(legacy_sync_function, "b"),
        asyncio.to_thread(legacy_sync_function, "c"),
    )
    print(f"混合并发: {results}")

asyncio.run(demo())
```

### 常见陷阱：阻塞事件循环

```python-run
import asyncio, time

async def bad() -> None:
    """错误: time.sleep 阻塞整个事件循环"""
    print("错误: 开始 time.sleep")
    time.sleep(0.5)   # ← 整个事件循环卡死 0.5s！
    print("错误: time.sleep 结束")

async def good() -> None:
    """正确: asyncio.sleep 释放控制权"""
    print("正确: 开始 asyncio.sleep")
    await asyncio.sleep(0.5)  # ← 释放控制权，其他协程可以运行
    print("正确: asyncio.sleep 结束")

async def demo() -> None:
    # 错误示范: 两个 bad() 串行执行
    start = time.perf_counter()
    await asyncio.gather(bad(), bad())
    print(f"time.sleep: 总耗时 {time.perf_counter() - start:.2f}s (约 1.0s)")

    # 正确示范: 两个 good() 真正并发
    start = time.perf_counter()
    await asyncio.gather(good(), good())
    print(f"asyncio.sleep: 总耗时 {time.perf_counter() - start:.2f}s (约 0.5s)")

asyncio.run(demo())
```

> **如果在 async 函数中看到 `time.sleep()`、`requests.get()`、`open("big_file").read()`——这都是 Bug。**
>
> 正确做法：
> - `time.sleep(n)` → `await asyncio.sleep(n)`
> - `requests.get(url)` → `await aiohttp.ClientSession().get(url)`
> - `open("big").read()` → `await aiofiles.open("big").read()`
> - 或者将它们扔进线程池：`await asyncio.to_thread(blocking_func, args)`

## Part E: asyncio vs threading vs multiprocessing 决策矩阵

### 直观对比

| 维度 | asyncio | threading | multiprocessing |
|------|---------|-----------|-----------------|
| 并发模型 | 单线程 + 协程 | 多线程 + GIL | 多进程 + 独立 GIL |
| 切换开销 | 纳秒级（用户态） | 微秒级（内核态） | 毫秒级（进程间） |
| 最大并发数 | 10 万+ | 数百到数千 | 数十到数百 |
| CPU 并行 | 不能 | 不能（GIL 限制） | 能 |
| 共享状态 | 天然安全（单线程） | 共享内存（需锁） | 不共享（IPC） |
| 调试难度 | 中（堆栈跟踪复杂） | 高（竞态条件难复现） | 中（进程隔离有利调试） |
| 学习曲线 | 高（async/await 传染） | 中（小心锁） | 低（同步代码直接跑） |

### C10k 问题: asyncio 的杀手场景

C10k 问题（10,000 并发连接）用 `threading` 怎么处理？每个连接一个线程：10,000 个线程 × 8 MB 栈内存 = 80 GB 内存。不可行。

用 `asyncio`：一个线程 + 一个事件循环。每个连接只是一个协程对象（~几 KB）。10,000 个连接占用的内存不到 10 MB 附加开销。

```python
# asyncio 处理 10,000 连接的伪代码
import asyncio

async def handle_client(reader, writer):
    data = await reader.read(1024)
    writer.write(data)  # echo 回去
    await writer.drain()
    writer.close()

async def main():
    server = await asyncio.start_server(handle_client, '0.0.0.0', 8888)
    await server.serve_forever()

# asyncio.run(main())
# 一个线程，轻松处理上万并发连接
```

### 混合方案：asyncio + ProcessPoolExecutor

```python-run
import asyncio
from concurrent.futures import ProcessPoolExecutor

def cpu_heavy(data: int) -> int:
    """CPU 密集型 —— 必须在进程池中运行"""
    total = 0
    for i in range(data * 1_000_000):
        total += i * i
    return total

async def demo() -> None:
    # 主事件循环处理 I/O，CPU 密集任务委托给进程池
    loop = asyncio.get_event_loop()
    with ProcessPoolExecutor(max_workers=4) as pool:
        tasks = [
            loop.run_in_executor(pool, cpu_heavy, n)
            for n in [1, 2, 3, 4]
        ]
        results = await asyncio.gather(*tasks)
        print(f"CPU 任务结果: {results[:2]}...")  # 只打印前两个

asyncio.run(demo())
```

### 最终决策树

```
你的应用是什么类型？
├── Web 服务 / API 网关 / 即时通讯
│   → asyncio（FastAPI, aiohttp, sanic）
│
├── 数据处理管道 / 科学计算 / 图像处理
│   → multiprocessing 或 concurrent.futures.ProcessPoolExecutor
│
├── 传统后端（Django / Flask）+ 不追求极限并发
│   → threading + WSGI 服务器（Gunicorn 多 worker）
│
├── 同时需要高并发 I/O 和 CPU 计算
│   → asyncio 主循环 + ProcessPoolExecutor（CPU 任务子进程）
│
└── 简单脚本 / 一次性批处理
    → concurrent.futures.ThreadPoolExecutor（代码最简洁）
```

### 三条终极建议

1. **新项目用 asyncio 作为默认选择**。Python 生态正在全面转向 async：FastAPI、httpx、SQLAlchemy 2.0 async、aiohttp、aiofiles。即使你现在不需要 10,000 并发，async 的代码结构（显式 `await` = 明确的 I/O 点）也比线程（隐式上下文切换）更容易推理。

2. **如果代码已经是同步的，不要为了 async 而重写**。用 `asyncio.to_thread()` 或 `ThreadPoolExecutor` 把同步代码包裹起来。渐进式迁移 > 一次重写。

3. **CPU 密集型任务永远用多进程**。这是 Python 绕开 GIL 的唯一方法（在 free-threaded 构建成熟之前）。

## 本章小结

- **协程本质**：协程是协作式调度，通过 `await` 主动让出控制权；切换开销为纳秒级（函数调用级别），单线程可支撑数万并发；事件循环的核心是"队列 + `send(None)` 轮转 + I/O 多路复用"。
- **Task 与 TaskGroup**：`create_task` 实现并发（同时运行），`await` 是串行等待；`TaskGroup` (3.11+) 提供结构化并发，某个任务失败时自动取消兄弟任务，配合 `except*` 处理 `ExceptionGroup`。
- **最新语法 (3.11-3.13)**：`asyncio.to_thread()` 桥接同步阻塞代码，`eager_task_factory` 减少调度延迟，`asyncio.timeout()` 上下文管理器提供灵活的超时控制。
- **实战模式**：`asyncio.Queue` 实现生产者-消费者并支持背压控制，`asyncio.Semaphore` 实现速率限制，`asyncio.Event` 实现优雅关闭，`asyncio.to_thread()` 实现混合同步/异步代码。
- **决策指南**：asyncio 是 I/O 密集型高并发场景的默认选择（Web 服务、API 网关），CPU 密集任务通过 `ProcessPoolExecutor` 配合使用，同步遗留代码用 `to_thread()` 渐进式迁移而非全部重写。
