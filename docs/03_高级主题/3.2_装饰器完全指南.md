# 装饰器（Decorator）完全指南

> 关于装饰器的基础和进阶用法，参见 [装饰器深度解析](../01_基础/decorators_deep.md)。本章聚焦于更低层的机制、更高级的模式，以及在实际工程中的关键决策。

---

## Part A: 装饰器的底层机制

### 反糖（Desugaring）：`@` 到底做了什么

装饰器语法 `@decorator` 是纯粹的语法糖（Syntactic Sugar）。Python 编译器在**定义时**将其展开为赋值语句：

```python
@decorator
def func(): ...
```

等价于：

```python
def func(): ...
func = decorator(func)  # 在定义时执行，只有一次
```

关键点：**`decorator(func)` 在函数定义所在的模块/类命名空间中被执行**，不是在调用 `func()` 时才执行。这意味着装饰器的逻辑在 import 时就已经跑完了。

> 📖 官方文档：[Function definitions — Decorators](https://docs.python.org/3/reference/compound_stmts.html#function-definitions)

### 字节码视角：编译器生成了什么

用 `dis` 模块可以直接看到装饰器语法对应的字节码指令：

```python-run
import dis

def deco(f): return f

src = """
@deco
def f(): pass
"""
print("模块字节码:")
dis.dis(compile(src, "", "exec"))
```

模块字节码中的关键指令序列：

```
LOAD_NAME    deco          # 加载装饰器对象
LOAD_CONST   f             # 加载函数对象 (code object 已编译)
CALL         ...           # 调用 deco(f)
STORE_NAME   f             # 将返回值重新绑定到名字
```

**装饰器表达式本身也可以是任意表达式：**

```python-run
registry = []
def register(f):
    registry.append(f)
    return f

@register
def a(): pass

print(len(registry))  # 1 — import 时就已经注册
```

### import 时执行：为什么装饰器里不能做重活

由于装饰器在 import 时执行，耗时操作应该放在 wrapper 内，而非装饰器外层：

```python-run
# ❌ 错误: 在装饰器定义层就连接了数据库
def bad_cache(func):
    conn = connect_database()  # import 时就连接! 可能还没配置好!
    ...

# ✅ 正确: 延迟到首次调用
def good_cache(func):
    _conn = None
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        nonlocal _conn
        if _conn is None:
            _conn = connect_database()
        ...
    return wrapper
```

### `@wraps` 的内部实现：`update_wrapper` 源码分析

`functools.wraps` 本身也是装饰器，它是对 `functools.update_wrapper` 的 `partial` 封装：

```python
# functools.wraps 的简化等价:
def wraps(wrapped):
    return partial(update_wrapper, wrapped=wrapped)
```

`update_wrapper` 的核心逻辑：

```python
# 简化版 update_wrapper
def update_wrapper(wrapper, wrapped):
    wrapper.__name__ = wrapped.__name__
    wrapper.__qualname__ = wrapped.__qualname__
    wrapper.__doc__ = wrapped.__doc__
    wrapper.__dict__.update(getattr(wrapped, '__dict__', {}))
    wrapper.__wrapped__ = wrapped      # ← 关键: 链式引用
    return wrapper
```

`__wrapped__` 属性建立了装饰器的**链式引用**。当你 `@wraps(func)` 时，可以通过 `wrapper.__wrapped__` 一路追溯到最底层的原始函数：

```python-run
from functools import wraps

def deco1(f):
    @wraps(f)
    def w(*a, **k): return f(*a, **k)
    return w

def deco2(f):
    @wraps(f)
    def w(*a, **k): return f(*a, **k)
    return w

@deco1
@deco2
def original(): pass

# 沿着 __wrapped__ 可以走回 origin
print(original.__name__)           # original
print(original.__wrapped__.__name__)  # original
# 解包装工具 (Python 3.2+)
import inspect
print(inspect.unwrap(original) is original)  # True
```

> 📖 官方文档：[functools.update_wrapper](https://docs.python.org/3/library/functools.html#functools.update_wrapper), [functools.wraps](https://docs.python.org/3/library/functools.html#functools.wraps)

## Part B: 高级模式

### 可选参数装饰器：`@log` 或 `@log(level="DEBUG")`

核心技巧：检测第一个参数是否为**可调用对象（Callable）**。如果不是（即带括号调用），说明用户传了配置参数：

```python-run
import functools

def log(func=None, *, level="INFO"):
    def decorator(f):
        @functools.wraps(f)
        def wrapper(*args, **kwargs):
            print(f"[{level}] {f.__name__}")
            return f(*args, **kwargs)
        return wrapper
    if func is None:
        return decorator      # @log(level="DEBUG") → 返回 decorator
    else:
        return decorator(func) # @log → 直接装饰

@log
def foo(): return "foo"

@log(level="DEBUG")
def bar(): return "bar"

foo()  # [INFO] foo
bar()  # [DEBUG] bar
```

> 这种模式在 click、flask 等框架的装饰器 API 中大量使用。

### 带状态的装饰器：调用统计

当装饰器需要维护状态（计数、统计、限流计数），类装饰器比闭包更清晰：

```python-run
import functools, time

def timing_stats(func):
    stats = []
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        t = time.perf_counter()
        result = func(*args, **kwargs)
        stats.append(time.perf_counter() - t)
        return result
    wrapper.stats = stats                              # 暴露属性
    wrapper.report = lambda: f"avg={sum(stats)/len(stats):.6f}s"  # 暴露方法
    return wrapper

@timing_stats
def work(): time.sleep(0.01)
```

### 修改函数参数的装饰器

装饰器不仅能在函数前后加逻辑，还能**修改传入的参数**：

```python-run
# 类型强制转换
def coerce_types(func):
    hints = func.__annotations__
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        params = list(hints.items())
        args = list(args)
        for i, (name, hint) in enumerate(params[:len(args)]):
            if not isinstance(args[i], hint):
                args[i] = hint(args[i])
        return func(*args, **kwargs)
    return wrapper

@coerce_types
def add(a: int, b: int): return a + b

print(add("3", "4"))  # 7 — 自动转 int
```

```python-run
# 默认值注入 (依赖注入的基石)
def inject_defaults(**defaults):
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            for key, val in defaults.items():
                kwargs.setdefault(key, val)
            return func(*args, **kwargs)
        return wrapper
    return decorator

@inject_defaults(timeout=30, retries=3)
def fetch(url, timeout, retries):
    return f"GET {url} timeout={timeout} retries={retries}"

print(fetch("/api"))  # GET /api timeout=30 retries=3
```

### 修改类的类装饰器

类装饰器接收类、返回类（或别的对象）。用它来**批量修改类行为**比元类更简洁：

```python-run
# 自动生成 __repr__
def auto_repr(cls):
    def __repr__(self):
        items = ", ".join(f"{k}={v!r}" for k, v in self.__dict__.items())
        return f"{cls.__name__}({items})"
    cls.__repr__ = __repr__
    return cls

@auto_repr
class Point:
    def __init__(self, x, y): self.x = x; self.y = y

print(Point(1, 2))  # Point(x=1, y=2)
```

```python-run
# 字段约束装饰器
def validate(**validators):
    def decorator(cls):
        orig = cls.__init__
        @functools.wraps(orig)
        def new_init(self, *args, **kwargs):
            orig(self, *args, **kwargs)
            for field, check in validators.items():
                val = getattr(self, field)
                if not check(val):
                    raise ValueError(f"{field}={val!r}")
        cls.__init__ = new_init
        return cls
    return decorator

@validate(age=lambda v: 0 <= v <= 150)
class Person:
    def __init__(self, name, age):
        self.name = name; self.age = age
```

### `@singledispatch`：基于类型的函数重载

Python 没有原生函数重载，但 `functools.singledispatch` 可以按第一个参数的类型分发：

```python-run
from functools import singledispatch

@singledispatch
def describe(obj):
    return f"未知类型: {type(obj).__name__}"

@describe.register(int)
def _(obj):
    return f"整数 {obj}"

@describe.register(list)
def _(obj):
    return f"列表 ({len(obj)}元素)"

print(describe(42))      # 整数 42
print(describe([1,2]))   # 列表 (2元素)
print(describe(3.14))    # 未知类型: float
```

> `register` 返回的注册函数名无关紧要（习惯用 `_`），真正生效的是 `@describe.register` 的参数类型注解。

类方法中使用 `singledispatchmethod`：

```python-run
from functools import singledispatchmethod

class Formatter:
    @singledispatchmethod
    def format(self, value):
        raise NotImplementedError

    @format.register(int)
    def _(self, value):
        return hex(value)

fmt = Formatter()
print(fmt.format(255))  # 0xff
```

> 📖 官方文档：[functools.singledispatch](https://docs.python.org/3/library/functools.html#functools.singledispatch), [PEP 443](https://peps.python.org/pep-0443/)

## Part C: 装饰器与元类

### 什么时候用谁

| | 装饰器 | 元类 |
|---|--------|------|
| 作用范围 | 单个函数/类 | 整个类的全部实例和子类 |
| 执行时机 | 定义时（一次） | 类创建时及每个子类创建时 |
| 能否拦截属性访问 | 否（需配合描述符） | 是 (`__getattribute__`) |
| 能否修改类继承结构 | 否 | 是 |
| 典型场景 | 日志、计时、注册 | ORM、序列化框架、ABC |

**经验法则**：
- **装饰器** = 给某个具体函数/类 "加个功能"
- **元类** = 定义 "某一类类的行为规则"
- 如果你发现自己要给很多类加同样的装饰器，考虑 `__init_subclass__` 或元类

### `__init_subclass__` vs 类装饰器 vs 元类

三者的选择逻辑：

```
你要对所有子类做同一件事？
  ├── 只需要在子类**定义时**介入？
  │     └── __init_subclass__  （最轻量）
  ├── 需要在子类**实例化时**介入？
  │     └── 元类
  └── 只需要改一个特定的类？
        └── 类装饰器  （最直观）
```

```python-run
# __init_subclass__: 在子类定义时自动调用
class Base:
    _registry = {}
    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        cls._registry[cls.__name__] = cls

class Dog(Base): pass
class Cat(Base): pass

print(Base._registry)  # {'Dog': <class 'Dog'>, 'Cat': <class 'Cat'>}
```

> 📖 官方文档：[__init_subclass__](https://docs.python.org/3/reference/datamodel.html#object.__init_subclass__), [Metaclasses](https://docs.python.org/3/reference/datamodel.html#metaclasses)

## Part D: 标准库中的装饰器——内部机制

### `@functools.lru_cache`：不仅仅是缓存

`lru_cache` 在装饰器内部维护一个**有序字典（OrderedDict）**，以 `(args, kwargs)` 的哈希为键。关键参数：

| 参数 | 含义 | 默认值 |
|------|------|--------|
| `maxsize` | 最大缓存条目数 | 128 |
| `typed` | 是否区分类型（`f(3)` vs `f(3.0)`） | False |

```python-run
from functools import lru_cache

@lru_cache(maxsize=128, typed=False)
def fib(n):
    return n if n < 2 else fib(n-1) + fib(n-2)

print(fib(30))
print(fib.cache_info())  # CacheInfo(hits=28, misses=31, maxsize=128, currsize=31)
fib.cache_clear()        # 清空缓存
```

内部近似实现（使用 `OrderedDict` + LRU 淘汰）：

```python
# lru_cache 简化版核心思路
def lru_cache_simple(maxsize=128):
    def decorator(func):
        cache = OrderedDict()
        @wraps(func)
        def wrapper(*args, **kwargs):
            key = args + tuple(kwargs.items())
            if key in cache:
                cache.move_to_end(key)    # 最近使用 → 移到末尾
                return cache[key]
            result = func(*args, **kwargs)
            cache[key] = result
            if len(cache) > maxsize:
                cache.popitem(last=False) # FIFO 淘汰
            return result
        return wrapper
    return decorator
```

> 📖 官方文档：[functools.lru_cache](https://docs.python.org/3/library/functools.html#functools.lru_cache)

### `@functools.cached_property`：基于描述符的惰性计算

`cached_property` 是一个**非数据描述符（Non-data Descriptor）**。首次访问时计算值，然后**直接写入实例的 `__dict__`**，后续访问走实例字典，不再调用描述符：

```python-run
from functools import cached_property

class Stats:
    def __init__(self, data):
        self.data = data

    @cached_property
    def variance(self):
        print("  > 正在计算...")
        avg = sum(self.data) / len(self.data)
        return sum((x - avg)**2 for x in self.data) / len(self.data)

s = Stats([1., 2., 3.])
print(s.variance)  # 计算并打印
print(s.variance)  # 直接从 __dict__ 取值，不打印
```

> 💡 与 `property` 的关键区别：`property` 每次访问都调用 getter；`cached_property` 只在第一次调用，之后值被存储在实例的 `__dict__` 中，跟普通属性一样快。

> 📖 官方文档：[functools.cached_property](https://docs.python.org/3/library/functools.html#functools.cached_property)

### `@functools.total_ordering`：自动补齐比较方法

只需定义 `__eq__` 和**任意一个**比较方法（`__lt__`/`__le__`/`__gt__`/`__ge__`），`@total_ordering` 自动生成其余三个：

```python-run
from functools import total_ordering

@total_ordering
class Version:
    def __init__(self, major, minor):
        self.major = major
        self.minor = minor

    def __eq__(self, other):
        return (self.major, self.minor) == (other.major, other.minor)

    def __lt__(self, other):
        return (self.major, self.minor) < (other.major, other.minor)

v1, v2 = Version(1, 0), Version(2, 0)
print(v1 <= v2, v1 > v2, v1 != v2)  # 全部可用
```

> ⚠️ 性能提示：自动生成的比较方法效率比手写略低（需要额外的方法调用）。对于热路径代码，手动实现所有比较方法更优。

> 📖 官方文档：[functools.total_ordering](https://docs.python.org/3/library/functools.html#functools.total_ordering)

### `@dataclass`：终极类装饰器

`@dataclass` 是一个类装饰器，它在类定义后动态生成 `__init__`、`__repr__`、`__eq__` 等方法：

```python
# @dataclass 生成的方法大致等价于:
def dataclass(cls):
    cls.__init__ = _make_init(cls)    # 根据字段生成 __init__
    cls.__repr__ = _make_repr(cls)    # 根据字段生成 __repr__
    cls.__eq__   = _make_eq(cls)      # 根据字段生成 __eq__
    # 可选: __hash__, __lt__, ...
    return cls
```

核心机制：`dataclass` 在类体中**查找带有类型注解但没有赋值的变量名**（以及使用 `field()` 赋默认值的变量名），将它们识别为字段（Field），然后动态生成方法。

```python-run
from dataclasses import dataclass, field, fields

@dataclass
class Product:
    name: str
    price: float = 0.0
    tags: list = field(default_factory=list)

p = Product("Book", 29.9)
print(p)                    # Product(name='Book', price=29.9, tags=[])
print(fields(Product))      # 完整的字段元信息
```

> 📖 官方文档：[dataclasses](https://docs.python.org/3/library/dataclasses.html), [PEP 557](https://peps.python.org/pep-0557/)

## Part E: 常见陷阱与决策

### 陷阱 1：装饰器改签名，调用方崩溃

```python-run
def takes_callable(f):
    return f  # 返回原函数，签名为 __init__(self, ...)

@takes_callable
class MyHandler:
    def __init__(self, config):
        self.config = config

# 用户代码期望实例化
try:
    h = MyHandler({"debug": True})  # TypeError: takes_callable() takes 1 arg ...
except TypeError as e:
    print(f"错误: {e}")
    # 类被装饰器替换成了函数，无法实例化!
```

**规则**：装饰函数返回函数，装饰类返回类。装饰类时如果想返回别的东西（注册、包装），确保不影响原始的使用方式。

### 陷阱 2：装饰器顺序 = 执行顺序

这是最容易出错的细节之一：

```python-run
def upper(func):
    @functools.wraps(func)
    def w(*a, **kw): return func(*a, **kw).upper()
    return w

def exclaim(func):
    @functools.wraps(func)
    def w(*a, **kw): return func(*a, **kw) + "!!!"
    return w

@upper       # 第二步: upper(exclaim_wrapper)
@exclaim     # 第一步: exclaim(hello)
def hello(): return "hello"

print(hello())  # HELLO!!!
# 先加 !!!，再变大写
```

装饰时**从下往上**，调用时**从上往下**（洋葱模型）。如果两个装饰器的行为会互相影响，一定要测试最终的调用顺序是否符合预期。

### 陷阱 3：方法装饰器丢失 `self`

当装饰器被用在**方法**（而不是普通函数）上时，需要注意：

```python-run
# ❌ 错误: wrapper 是普通函数，没有 self
def broken_log(func):
    def wrapper(*args, **kwargs):  # 没有接收 self 的位置
        print(f"调用 {func.__name__}")
        return func(*args, **kwargs)
    return wrapper

class Service:
    @broken_log
    def work(self): pass  # work 不再是绑定方法

# ✅ 正确: wrapper 用 *args/**kwargs 透传第一个 self 参数
def safe_log(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        print(f"调用 {func.__name__}")
        return func(*args, **kwargs)
    return wrapper

# 使用 *args 透传时, 第一个 args[0] 自然地就是 self
```

> **更稳妥的方案**：如果需要批量给类方法加装饰器，用类装饰器遍历 `cls.__dict__` 给每个方法加装饰，而不是单独在每个方法上用 `@`。

## 本章小结

- **底层机制**：`@decorator` 在定义时展开为 `func = decorator(func)`，执行时机在 import 时而非调用时；`@wraps` 通过 `__wrapped__` 建立链式引用，`inspect.unwrap()` 可追溯到原始函数。
- **高级模式**：可选参数装饰器通过检测首个参数是否为 callable 区分 `@log` 和 `@log(level="DEBUG")`；类装饰器批量修改类行为，`__init_subclass__` 在子类定义时自动介入。
- **标准库装饰器**：`lru_cache` 用有序字典实现 LRU 淘汰，`cached_property` 是非数据描述符（计算后写入 `__dict__`），`total_ordering` 自动补齐比较方法，`dataclass` 动态生成 `__init__`/`__repr__`/`__eq__`。
- **常见陷阱**：装饰器应保持签名（用 `@wraps`），装饰器顺序从下往上装饰、从上往下调用，方法装饰器需用 `*args` 透传 `self`。
