# 3.7 内存管理与性能

Python 帮你管理内存，但**不是免费的**——引用计数、循环回收、对象分配都有成本。理解这些机制，你才能写出在百万级数据量下仍然丝滑的代码。

本章回答三个问题：**Python 如何自动管理内存**、**什么情况下内存不会被释放**、**如何测量和优化 Python 程序的内存与速度**。

## 7.1 引用计数：Python 内存管理的基石

### 它是什么

Python 中每个对象都有一个**引用计数**（Reference Count）——记录有多少个变量或容器引用了它。当引用计数降为 0，对象立即被销毁，内存被回收。这就是 Python **不需要手动 `free()`** 的根本原因。

> 📖 官方文档：[Reference Counting](https://docs.python.org/3/c-api/refcounting.html), [sys.getrefcount](https://docs.python.org/3/library/sys.html#sys.getrefcount)

```python-run
import sys

a = []
print(sys.getrefcount(a))  # 2 —— a 引用了一次，getrefcount 参数又引用了一次

b = a
print(sys.getrefcount(a))  # 3 —— a、b、getrefcount 参数，三个引用

c = [a]                    # c 列表内部引用了 a
print(sys.getrefcount(a))  # 4

del b
print(sys.getrefcount(a))  # 3 —— b 被删除了

del c
print(sys.getrefcount(a))  # 2 —— 只剩 a 和 getrefcount 的参数
```

**`sys.getrefcount()` 的 "+1" 陷阱**：`getrefcount(obj)` 本身会临时引用 `obj`，所以返回值总是比"你感觉的"多 1。记住：当 `getrefcount` 返回 2 时，意味着外部只有 1 个引用。

### 引用循环：引用计数的死穴

引用计数的致命缺陷：两个对象互相引用，计数永远不会降到 0。

```python-run
import sys

class Node:
    def __init__(self, name: str):
        self.name = name
        self.next: "Node | None" = None

a = Node("A")
b = Node("B")
a.next = b
b.next = a  # 循环引用！

print(f"a refs: {sys.getrefcount(a) - 1}")  # 2 (a + b.next 引用)
print(f"b refs: {sys.getrefcount(b) - 1}")  # 2 (b + a.next 引用)

# 即使删除外部引用，循环内的对象引用计数也不为 0
del a, b
# a 和 b 现在只有彼此相互引用——内存泄漏！
```

这正是 Python 需要**垃圾回收器（Garbage Collector）**的原因——引用计数处理不了循环引用，需要 GC 来兜底。

## 7.2 分代垃圾回收

### 它是什么

Python 的 GC 采用**分代回收（Generational Garbage Collection）**策略——对象按"存活时间"分到三个代（Generation），越年轻的代越频繁扫描。

| 代 | 包含对象 | 回收频率 | 触发阈值（默认） |
|----|---------|---------|----------------|
| gen0 | 新创建的对象 | 最频繁 | 700 |
| gen1 | 经历过一次回收仍存活 | 中等 | 10 |
| gen2 | 经历过两次回收仍存活 | 最稀疏 | 10 |

**晋升规则**：对象在一次 GC 扫描中存活下来，就会晋升（Promote）到下一代。gen0 扫得勤快（大多数对象都是短命的），gen2 扫得从容（长命对象很少产生循环引用）。

```python-run
import gc

print(f"GC 启用: {gc.isenabled()}")
print(f"阈值:    {gc.get_threshold()}")    # (700, 10, 10)

# gc.get_count() 返回：(gen0_allocations, gen1_allocations, gen2_allocations)
print(f"当前计数: {gc.get_count()}")

# 手动触发回收
unreachable = gc.collect()
print(f"回收数量: {unreachable}")
```

### 调优阈值

```python-run
import gc

# 提高阈值 = 减少 GC 扫描频率 = 吞吐量更高但瞬时内存使用可能更高
gc.set_threshold(2000, 15, 15)

# 极端情况：完全禁用 GC（仅在确定没有循环引用时用）
# gc.disable()     # 禁用后循环引用会变成真正的内存泄漏
# gc.enable()      # 重新启用

# 生产环境中更常见的做法：
# 1. 保持 GC 开启（默认值对大多数程序都合理）
# 2. 在处理大批量短命对象后手动触发一次
for _ in range(100000):
    obj = {"data": _}  # 这些对象很快变垃圾
gc.collect()            # 立刻清理，不用等阈值
```

> ⚠️ **不要轻易禁用 GC**。游戏引擎、金融计算等场景中，如果你**确认**没有循环引用，禁用 GC 可以避免间歇性的 GC 停顿——但这是有代价的：一旦意外出现了循环引用，就会内存泄漏。

> 📖 官方文档：[gc — Garbage Collector interface](https://docs.python.org/3/library/gc.html), [gc.set_threshold](https://docs.python.org/3/library/gc.html#gc.set_threshold)

## 7.3 weakref：打破循环引用

### 它是什么

**弱引用（Weak Reference）** 是一种"不增加引用计数"的引用。当你需要引用一个对象但不想阻止它被回收时，用弱引用。

### 基本用法

```python-run
import weakref
import sys

class Data:
    def __init__(self, name):
        self.name = name

d = Data("important")
r = weakref.ref(d)          # r 是 d 的弱引用

print(f"r()  = {r()}")       # <__main__.Data object> —— 获取被引用对象
print(f"refcount: {sys.getrefcount(d)}")  # 2 (d + getrefcount 参数)
# 注意：r 没有增加引用计数！

del d                        # 删除唯一强引用
print(f"r()  = {r()}")       # None —— 对象已被回收
```

### WeakValueDictionary：会自动"瘦身"的缓存

```python-run
import weakref

class Image:
    def __init__(self, key: str, data: bytes):
        self.key = key
        self.data = data
        print(f"  + 加载 {key} ({len(data)} bytes)")

    def __del__(self):
        print(f"  - 卸载 {self.key}")

# 普通 dict 作为缓存：值永远不被回收（内存泄漏）
# cache = {}                    # 危险！

# WeakValueDictionary：当值没有被外部引用时，自动清除条目
cache = weakref.WeakValueDictionary()

img = Image("hero.png", b"x" * 1024 * 1024)   # 1MB 图片
cache["hero"] = img
print(f"缓存中有: {list(cache.keys())}")       # ['hero']

del img  # 删除唯一强引用
print(f"缓存中: {list(cache.keys())}")         # [] —— 自动清理！
```

`WeakValueDictionary` 的典型应用：缓存管理器——你希望缓存能提升性能，但又不想缓存阻止对象被 GC 回收。当原始对象不再被使用，缓存条目自动消失，不会造成内存泄漏。

### WeakKeyDictionary

```python-run
import weakref

# WeakKeyDictionary：键是弱引用
# 用途：给对象附加元数据，但不阻止对象被回收
metadata = weakref.WeakKeyDictionary()

class Connection:
    def __init__(self, conn_id: int):
        self.conn_id = conn_id
    def __del__(self):
        print(f"  连接 {self.conn_id} 已关闭")

conn = Connection(42)
metadata[conn] = {"last_query": "SELECT *", "duration_ms": 15}

print(f"元数据: {metadata[conn]}")
del conn  # Connection 对象被回收
# metadata 中的对应条目自动消失
print(f"元数据条目数: {len(metadata)}")  # 0
```

### weakref.finalize：优雅的清理回调

```python-run
import weakref

class ManagedFile:
    def __init__(self, path: str):
        self.path = path
        self._file = open(path, 'w')
        # 注册清理函数——对象被回收时自动调用
        weakref.finalize(self, self._close_file, self.path)

    @staticmethod
    def _close_file(path):
        print(f"[GC 清理] 关闭文件: {path}")
        # 注意：此时 self 可能已被部分销毁，所以用 staticmethod + 传 path

f = ManagedFile("/tmp/test.log")
print("f 存在……")
# 当 f 被回收时，finalize 回调自动触发
del f
print("f 已被删除")
```

> 📖 官方文档：[weakref — Weak references](https://docs.python.org/3/library/weakref.html), [weakref.finalize](https://docs.python.org/3/library/weakref.html#weakref.finalize)

## 7.4 `__slots__` 的内存分析

### 内存布局差异

```python-run
import sys

class WithDict:
    def __init__(self, x, y, z):
        self.x = x
        self.y = y
        self.z = z

class WithSlots:
    __slots__ = ('x', 'y', 'z')
    def __init__(self, x, y, z):
        self.x = x
        self.y = y
        self.z = z

d = WithDict(1, 2, 3)
s = WithSlots(1, 2, 3)

# 实例本身的大小
print(f"实例 - WithDict:  {sys.getsizeof(d)} bytes")
print(f"实例 - WithSlots: {sys.getsizeof(s)} bytes")
# __dict__ 的额外开销（通常 ~104 bytes 起步）
print(f"__dict__ 大小:    {sys.getsizeof(d.__dict__)} bytes")
print(f"总共 - WithDict:  {sys.getsizeof(d) + sys.getsizeof(d.__dict__)} bytes")

# WithSlots 没有 __dict__
print(f"有 __dict__:      {hasattr(s, '__dict__')}")
```

**`__slots__` 省了多少？** 对于只有 3 个属性的小对象，一个 `__dict__` 实例约 56 + 104 = **~160 bytes**，而 `__slots__` 实例只约 **56 bytes**——节省约 65%。当你有 100 万个实例时，差距是 160MB vs 56MB。

### 什么时候值得用 `__slots__`

| 场景 | 结论 |
|------|------|
| 创建数百万个小对象（坐标点、事件、像素） | **值得**。内存节省巨大。 |
| 普通业务对象（几十个实例） | **不值得**。代码复杂度增加，收益微不足道。 |
| 需要动态添加属性 | **不能用**。`__slots__` 移除了 `__dict__`。 |
| 框架基类 | **慎用**。子类会被迫也定义 `__slots__`，否则自动获得 `__dict__`。 |
| 需要 `pickle` / `copy` 兼容 | **注意**。`__slots__` 对象需要定义 `__getstate__` 才能被 pickle。 |

### `@dataclass(slots=True)` —— 现代首选

Python 3.10+ 的 dataclasses 原生支持 `slots=True`：

```python-run
from dataclasses import dataclass

@dataclass(slots=True)
class Vector3:
    x: float
    y: float
    z: float

v = Vector3(1.0, 2.0, 3.0)
print(v)                          # Vector3(x=1.0, y=2.0, z=3.0)
print(f"有 __dict__: {hasattr(v, '__dict__')}")  # False

# 对比：不使用 slots
@dataclass
class Vector3Dict:
    x: float
    y: float
    z: float

import sys
vs = Vector3(1.0, 2.0, 3.0)
vd = Vector3Dict(1.0, 2.0, 3.0)
print(f"slots=True:  {sys.getsizeof(vs)} bytes")
print(f"slots=False: {sys.getsizeof(vd) + sys.getsizeof(vd.__dict__)} bytes")
```

> `@dataclass(slots=True)` 是目前创建内存高效的小对象的**最佳实践**——一行代码，自动生成 `__init__`、`__repr__`、`__slots__`。

> 📖 官方文档：[dataclasses](https://docs.python.org/3/library/dataclasses.html), [__slots__](https://docs.python.org/3/reference/datamodel.html#slots)

## 7.5 tracemalloc：追踪内存分配

### 它是什么

`tracemalloc` 是 Python 3.4+ 内置的内存追踪模块。它能回答：**哪些代码分配了最多的内存**、**两个时间点之间多了哪些内存**、**哪里可能产生了内存泄漏**。

### 找出当前内存大户

```python-run
import tracemalloc

tracemalloc.start()

# 分配一些内存
data1 = [bytearray(1024) for _ in range(100)]
data2 = "x" * (1024 * 1024)       # 1 MB 字符串

snapshot = tracemalloc.take_snapshot()
top_stats = snapshot.statistics('lineno')

print("Top 5 内存分配:")
for stat in top_stats[:5]:
    print(f"  {stat}")
```

### 对比快照：找到内存增量的来源

```python-run
import tracemalloc

tracemalloc.start()

snap_before = tracemalloc.take_snapshot()

# 模拟操作：分配一批对象
cache = {}
for i in range(1000):
    cache[f"key_{i}"] = bytearray(256)

# 模拟操作：释放一部分
for i in range(500):
    del cache[f"key_{i}"]

snap_after = tracemalloc.take_snapshot()

# 对比差异
stats = snap_after.compare_to(snap_before, 'lineno')
print("内存增量 Top 3:")
for stat in stats[:3]:
    print(f"  {stat}")
```

### 实战：找到内存泄漏

```python-run
import tracemalloc

class LeakyCache:
    """危险！缓存永不清理——内存泄漏"""
    _cache: dict[str, bytes] = {}

    @classmethod
    def get(cls, key: str) -> bytes:
        if key not in cls._cache:
            cls._cache[key] = key.encode() * 1024  # 模拟计算
        return cls._cache[key]


tracemalloc.start(10)  # 保留 10 帧调用栈

snap1 = tracemalloc.take_snapshot()
for i in range(5000):
    LeakyCache.get(f"req_{i}")
snap2 = tracemalloc.take_snapshot()

stats = snap2.compare_to(snap1, 'lineno')
for stat in stats[:3]:
    print(f"  {stat}")
    # 会精确告诉你：哪个文件、第几行、分配了多少内存
```

> 📖 官方文档：[tracemalloc — Trace memory allocations](https://docs.python.org/3/library/tracemalloc.html)

## 7.6 dis：查看字节码

### 它是什么

Python 执行代码前，会先将源码编译成**字节码（Bytecode）**——一种 Python 虚拟机可以逐条执行的中间指令。`dis` 模块让你窥探这个过程。

> 📖 官方文档：[dis — Disassembler for Python bytecode](https://docs.python.org/3/library/dis.html)

```python-run
import dis

def add(a, b):
    return a + b

dis.dis(add)
```

输出解读：

| 指令 | 含义 |
|------|------|
| `LOAD_FAST` | 从本地变量数组中读取变量 |
| `LOAD_GLOBAL` | 从全局命名空间中读取变量 |
| `LOAD_ATTR` | 读取对象属性（`obj.attr`） |
| `STORE_FAST` | 写入本地变量 |
| `CALL` | 调用函数 |
| `BINARY_OP` | 二元运算（Python 3.11+ 新增，替代 `BINARY_ADD` 等） |
| `RETURN_VALUE` | 返回栈顶值 |

### 理解执行过程

```python-run
import dis

def compute(x):
    y = x * 2
    return y + 10

dis.dis(compute)
```

逐行解读：
1. `LOAD_FAST x` —— 把 `x` 压入栈
2. `LOAD_CONST 2` —— 把常量 `2` 压入栈
3. `BINARY_OP *` —— 弹出两个值，做乘法，结果压回栈
4. `STORE_FAST y` —— 栈顶弹出，存入局部变量 `y`
5. `LOAD_FAST y` + `LOAD_CONST 10` + `BINARY_OP +` + `RETURN_VALUE` —— 同理

Python 虚拟机是**栈式虚拟机**（Stack-based VM）——所有计算通过一个求值栈（Evaluation Stack）完成。

### 为什么了解字节码有用：对比等价构造

```python-run
import dis

# 列表推导式
def squares_comp(n):
    return [x*x for x in range(n)]

# 等价 for 循环
def squares_loop(n):
    result = []
    for x in range(n):
        result.append(x * x)
    return result

print("=== 列表推导式 ===")
dis.dis(squares_comp)
print("\n=== for 循环 ===")
dis.dis(squares_loop)
```

核心差异：列表推导式有**专用的 `LIST_APPEND` 字节码**，绕过了 `.append()` 方法的属性查找（`LOAD_ATTR`）和函数调用（`CALL`）。这就是列表推导式更快的原因——它少做了两次字典查找。

### 常见性能陷阱的字节码证据

```python-run
import dis

# 反模式：循环内反复访问全局变量
def slow(items):
    for x in items:
        y = math.sqrt(x)   # LOAD_GLOBAL 'math' 每次循环都执行
        # ...

# 好模式：缓存到本地变量
def fast(items):
    sqrt = math.sqrt       # LOAD_GLOBAL 只执行一次
    for x in items:
        y = sqrt(x)        # LOAD_FAST —— 直接数组索引

# 对比字节码：
# LOAD_GLOBAL 需要哈希表查找
# LOAD_FAST 直接通过数组索引访问——快 1-2 个数量级
```

## 7.7 性能优化技巧

### 1. 循环中的局部变量缓存

```python-run
import math
import time

items = list(range(10_000_000))

# 每次循环都做 global lookup
start = time.perf_counter()
for x in items:
    _ = math.sqrt(x)       # LOAD_GLOBAL 两次（math + sqrt）
print(f"全局变量: {time.perf_counter() - start:.3f}s")

# 缓存到局部变量
start = time.perf_counter()
sqrt = math.sqrt           # 一次 LOAD_GLOBAL
for x in items:
    _ = sqrt(x)            # LOAD_FAST
print(f"局部缓存: {time.perf_counter() - start:.3f}s")
```

### 2. `functools.lru_cache`：计算结果缓存

```python-run
import functools
import time

@functools.lru_cache(maxsize=128)
def fibonacci(n: int) -> int:
    if n < 2:
        return n
    return fibonacci(n - 1) + fibonacci(n - 2)

start = time.perf_counter()
result = fibonacci(200)
elapsed = time.perf_counter() - start
print(f"fibonacci(200) = {result}, 耗时: {elapsed:.6f}s")
print(f"缓存信息: {fibonacci.cache_info()}")

# 没有缓存时，时间复杂度 O(2^n)；有了 lru_cache → O(n)
# fibonacci(200) 没有缓存时宇宙毁灭也算不完
```

### 3. `__slots__` 的属性访问速度

`__slots__` 不仅省内存，也加速属性访问——因为 `member_descriptor` 通过数组索引直接读取，而 `__dict__` 需要哈希查找。

```python-run
import time

class WithDict:
    def __init__(self):
        self.x = 0

class WithSlots:
    __slots__ = ('x',)
    def __init__(self):
        self.x = 0

N = 10_000_000

d_obj = WithDict()
s_obj = WithSlots()

start = time.perf_counter()
for _ in range(N):
    _ = d_obj.x
print(f"__dict__ 访问:  {time.perf_counter() - start:.3f}s")

start = time.perf_counter()
for _ in range(N):
    _ = s_obj.x
print(f"__slots__ 访问: {time.perf_counter() - start:.3f}s")
```

### 4. `functools.cached_property`：实例级惰性缓存

Python 3.8+ 引入，比 `@property` + 手动缓存更优雅：

```python-run
import functools

class DataSet:
    def __init__(self, numbers: list[int]):
        self.numbers = numbers

    @functools.cached_property
    def avg(self):
        print("  (计算中...)")
        return sum(self.numbers) / len(self.numbers)

ds = DataSet([1, 2, 3, 4, 5])
print(ds.avg)  # (计算中...) → 3.0
print(ds.avg)  # 3.0 —— 不重新计算
```

`cached_property` 的原理和前文 `LazyProperty` 一样——计算后将值写入 `instance.__dict__`。但它还处理了锁（线程安全）和 `__slots__` 兼容。

### 5. 优化全景对比

| 技术 | 优化什么 | 适用场景 | 代价 |
|------|---------|---------|------|
| 局部变量缓存 | 属性/全局访问速度 | 热循环内部 | 代码稍长 |
| `__slots__` | 内存 + 属性访问 | 大量小对象 | 不能动态添加属性 |
| `@lru_cache` | 重复计算 | 纯函数、递归 | 额外内存（缓存） |
| `@cached_property` | 惰性计算 | 对象属性，只算一次 | 属性不可变后 |
| `dataclass(slots=True)` | 内存 + 开发效率 | 数据载体类 | Python 3.10+ |

## 本章小结

- **引用计数与 GC**：Python 基于引用计数管理内存（引用归零即回收），但循环引用需要分代垃圾回收器（GC）兜底；GC 按对象存活时间分三代扫描，越年轻的代扫描越频繁。
- **弱引用**：`weakref` 提供不增加引用计数的引用方式，`WeakValueDictionary`/`WeakKeyDictionary` 实现"自动瘦身"缓存，`weakref.finalize` 注册对象回收时的清理回调。
- **`__slots__`**：用固定大小的槽位替代 `__dict__`，节省约 65% 内存并加速属性访问；`@dataclass(slots=True)` 是现代创建内存高效数据类的最佳实践。
- **性能分析工具**：`tracemalloc` 追踪内存分配和泄漏，`dis` 查看字节码解释性能差异（如列表推导式比 `for` 循环快的原因），局部变量缓存和 `lru_cache` 是两大常用优化手段。
- **优化全景**：局部变量缓存减少全局查找、`__slots__` 节省内存和访问时间、`@lru_cache` 缓存纯函数结果、`@cached_property` 惰性计算一次——每个技术都有明确的适用场景和代价。
