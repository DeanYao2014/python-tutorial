# 3.5 并发：线程与 GIL

"Python 的多线程是假的"——这句话你听过多少次？真相远比这句话复杂。本章从 GIL 的底层原因开始，遍历 `threading`、`multiprocessing`、`concurrent.futures` 三个模块，帮你建立 Python 并发选择的正确直觉。

## Part A: GIL 深度剖析

### GIL 是什么

**GIL（Global Interpreter Lock，全局解释器锁）** 是 CPython 内存管理中的一个互斥锁（Mutex）。它保证同一时刻**在一个进程内, 只有一个线程在执行 Python 字节码**。

> Python 多线程 = `并发（轮流切换）`，不是并行（同时跑）, 
> Python 的多线程（同一进程内），无论开多少个线程，永远只能跑满 1 个 CPU 核心。
> 线程 A 在跑 → 锁被占用 ; 线程 B、C、D 全部在等待锁 ; 它们不能同时在不同 CPU 核心上运行
```python-run
import time, threading
import os

def loop():
  print(f'thread {threading.current_thread().name} is running..')

  n = 0 
  while n < 5:
    n = n +1
    print(f'thread {threading.current_thread().name} → {n}: 当前进程: {os.getpid()}')
    time.sleep(1)
  print(f'thread {threading.current_thread().name} ended')


print(f'thread {threading.current_thread().name} is running')
t = threading.Thread(target=loop, name="LoopThread")    
t.start()
t.join()
print(f'thread {threading.current_thread().name} ended')
```

关键前提：
- **GIL 是 CPython 的实现细节，不是 Python 语言特性**。Jython 和 IronPython 没有 GIL。
- GIL 只保护 Python 对象的内部状态——你的业务逻辑不在 GIL 保护范围内。

```python-run
import sys

# sys.getswitchinterval() 返回 GIL 的切换间隔（默认 ~5ms）
print(f"GIL 切换间隔: {sys.getswitchinterval()} 秒")

# 可以修改（大多数时候不需要）
sys.setswitchinterval(0.01)  # 改成 10ms

# Python 3.13+ 可以检查 GIL 是否启用
if hasattr(sys, "_is_gil_enabled") and callable(sys._is_gil_enabled):
    import sys as sys_check
    # sys._is_gil_enabled()  # 返回 True（标准 CPython）
```

> 📖 官方文档：[Global Interpreter Lock](https://docs.python.org/3/glossary.html#term-global-interpreter-lock)

### 为什么需要 GIL：引用计数

CPython 使用**引用计数（Reference Counting）**来管理内存。每个 Python 对象内部有一个 `ob_refcnt` 字段，记录有多少个引用指向它。当引用计数归零时，对象被立即释放。

```python
# 如果不保护，多线程同时修改 refcnt 会导致：
# 情况 1: 计数过多 → 内存泄漏（对象永远不释放）
# 情况 2: 计数过少 → use-after-free（崩溃）
```

**加 GIL 是最简单的解决方案**——任何一个线程在执行 Python 代码之前，必须先获取 GIL。这样 `ob_refcnt` 的加减就天然安全了。

> 不用 GIL 的替代方案：每个对象单独加锁（内存开销太大）、无锁原子操作（CPython 在不同平台上实现起来太复杂）、分代内存管理配合 GC（JVM/.NET 的方案，但 CPython 已经绑定了引用计数的设计）。

### GIL 的行为

GIL 不是一直霸占着不放。在以下情况下，当前线程**释放 GIL**：

1. **执行 I/O 操作时**（文件读写、网络请求、`time.sleep()`）
2. **每隔 ~5ms**（`sys.getswitchinterval()`），解释器检查是否需要切换到其他线程
3. **调用 C 扩展中主动释放 GIL 的代码**（如 NumPy 的某些操作）

```python-run
import time, threading

def io_worker(name: str) -> None:
    print(f"[{name}] 开始")
    time.sleep(0.5)  # sleep 期间释放 GIL，其他线程可以运行
    print(f"[{name}] 完成")

# I/O 密集型：线程确实能并行（在 I/O 等待期间释放 GIL）
t1 = threading.Thread(target=io_worker, args=("A",))
t2 = threading.Thread(target=io_worker, args=("B",))
t1.start(); t2.start()
t1.join(); t2.join()
```

### CPU 密集型：GIL 的反面

但如果有两个线程都在做纯计算呢？

```python-run
import time, threading

def cpu_work(n: int) -> int:
    total = 0
    for i in range(n):
        total += i * i
    return total

N = 5_000_000

# 单线程
start = time.perf_counter()
cpu_work(N)
single = time.perf_counter() - start
print(f"单线程: {single:.3f}s")

# 双线程
start = time.perf_counter()
t1 = threading.Thread(target=cpu_work, args=(N,))
t2 = threading.Thread(target=cpu_work, args=(N,))
t1.start(); t2.start()
t1.join(); t2.join()
multi = time.perf_counter() - start
print(f"双线程: {multi:.3f}s (倍率 {multi/single:.1f}x)")
```

你猜对了吗？**双线程反而更慢**。因为在 CPU 密集型任务中，两个线程互相竞争 GIL：
- GIL 每 5ms 在线程间切换一次
- 切换本身有开销
- 两个线程永远不能真正并行
- 结果是：双线程 ~= 单线程的 1.1-1.3 倍耗时

> **核心结论**：GIL 让 CPU 密集型多线程比单线程**更慢**。如果你想用 Python 做并行计算，必须用多进程。

### Python 3.12/3.13 的无 GIL 之路

两个重要的 PEP 正在改变 GIL 的故事：

**PEP 684 (3.12)**：Per-Interpreter GIL —— 每个子解释器拥有独立的 GIL。这允许多个解释器实例在各自线程中**真正并行**运行（但共享数据的成本更高）。

**PEP 703 (3.13 preview)**：为 CPython 添加 `--disable-gil` 编译选项。这是一项巨大的工程——把引用计数改造成偏向引用计数（Biased Reference Counting），配合细粒度锁和延迟引用计数。目前作为实验性特性存在。

```python

# import sys
# sys._is_gil_enabled()  # 返回 False（在 nogil 构建中）
```

## Part B: `threading` 模块

### 线程的创建与 join

```python-run
import threading, time

def worker(name: str, delay: float) -> None:
    print(f"[{name}] 开始")
    time.sleep(delay)
    print(f"[{name}] 完成")

t1 = threading.Thread(target=worker, args=("A", 0.5))
t2 = threading.Thread(target=worker, args=("B", 0.3))
t1.start(); t2.start()
t1.join(); t2.join()  # 等待线程结束
print("全部完成")
```

`join()` 是同步点：调用 `join()` 的主线程会阻塞，直到目标线程结束。

### 线程安全：Lock 和 RLock

**没有 Lock 时**——竞态条件（Race Condition）：

```python-run
import threading

counter = 0

def increment() -> None:
    global counter
    for _ in range(100_000):
        temp = counter   # 读
        temp = temp + 1   # 改
        counter = temp    # 写 —— 这三步不是原子的！

threads = [threading.Thread(target=increment) for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()
print(f"无锁: counter = {counter} (期望 400000)")
```

> GIL 只保护**单个字节码操作**不被中断，不保护**多条字节码组成的逻辑操作**。`counter += 1` 编译成多条字节码，其中任何两条之间都可能被其他线程插队。

**用 Lock 保护临界区**：

```python-run
import threading

counter = 0
lock = threading.Lock()

def increment() -> None:
    global counter
    for _ in range(100_000):
        with lock:          # 获取锁
            counter += 1    # 现在安全了

threads = [threading.Thread(target=increment) for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()
print(f"有锁: counter = {counter}")
```

**Lock vs RLock**：

| | Lock | RLock |
|---|---|---|
| 同一线程重复获取 | 死锁 | 允许（记录次数） |
| 性能 | 更快 | 稍慢（需检查持有者） |
| 何时用 | 简单的互斥 | 嵌套函数调用、递归 |

```python
lock = threading.Lock()
with lock:
    with lock:  # 死锁！Lock 不支持可重入
        pass

rlock = threading.RLock()
with rlock:
    with rlock:  # OK——同一线程可重复获取
        pass
```

### Semaphore：限制并发数

```python-run
import threading, time

semaphore = threading.Semaphore(2)  # 同时最多 2 个线程

def limited(name: str) -> None:
    with semaphore:
        print(f"[{name}] 进入")
        time.sleep(0.2)
        print(f"[{name}] 离开")

for i in range(5):
    threading.Thread(target=limited, args=(f"T{i}",)).start()
# 始终只有 2 个线程在 "进入" 和 "离开" 之间
```

### threading.local()：线程私有的全局变量

```python-run
import threading

local = threading.local()

def set_and_print(name: str, value: int) -> None:
    local.value = value         # 设置线程本地变量
    print(f"[{name}] local.value = {local.value}")

t1 = threading.Thread(target=set_and_print, args=("T1", 10))
t2 = threading.Thread(target=set_and_print, args=("T2", 20))
t1.start(); t2.start()
t1.join(); t2.join()
# T1 输出 10，T2 输出 20 —— 互不干扰
```

### queue.Queue：线程间通信的标准方案

```python-run
import threading, queue, time

q: queue.Queue[int] = queue.Queue()

def producer() -> None:
    for i in range(3):
        time.sleep(0.1)
        q.put(i)
        print(f"生产: {i}")
    q.put(None)  # 毒丸：通知消费者结束

def consumer(name: str) -> None:
    while True:
        item = q.get()
        if item is None:
            q.put(None); break  # 放回毒丸，让其他消费者也能看到
        print(f"[{name}] 消费: {item}")

t_p = threading.Thread(target=producer)
t_c = threading.Thread(target=consumer, args=("A",))
t_p.start(); t_c.start()
t_p.join(); t_c.join()
```

### 线程什么时候真正有用？

GIL 让很多人对 Python 线程失去信心，但**线程在以下场景中非常有价值**：

1. **I/O 密集型任务**：网络请求、文件读写、数据库查询——线程在等待 I/O 时释放 GIL，其他线程可以继续运行
2. **需要共享状态的并发**：多个任务需要访问同一个数据结构，`threading` 直接共享内存，而 `multiprocessing` 需要进程间通信
3. **调用会释放 GIL 的 C 库**：NumPy、OpenCV 等 C 扩展在计算时释放 GIL，线程在此刻能真正并行

### 常见陷阱

```python-run
import threading

# 陷阱 1: 线程太多 → 上下文切换开销 > 收益
# 1000 个 I/O 线程看起来合理，但 CPU 在线程间切换的成本会急剧上升
# 解决: 用线程池限制并发数

# 陷阱 2: 主线程退出不管子线程
def daemon_worker() -> None:
    import time
    time.sleep(0.5)
    print("(这行可能永远不打印)")

t = threading.Thread(target=daemon_worker, daemon=True)
t.start()
print("主线程退出——daemon 线程被强制终止")
# daemon=True: 主线程退出时，子线程被强制终止，无法做清理
```

> 📖 官方文档：[threading](https://docs.python.org/3/library/threading.html)

## Part C: `multiprocessing` 模块

### 绕开 GIL 的正确姿势

```python-run
from multiprocessing import Process
from concurrent.futures import ProcessPoolExecutor
import time

def cpu_work(n: int) -> int:
    total = 0
    for i in range(n):
        total += i * i
    return total

N = 3_000_000

# 单进程
start = time.perf_counter()
cpu_work(N)
print(f"单进程: {time.perf_counter() - start:.3f}s")

# 4 进程并行（每个进程有独立的 GIL）
start = time.perf_counter()
with ProcessPoolExecutor(max_workers=4) as executor:
    futures = [executor.submit(cpu_work, N) for _ in range(4)]
    [f.result() for f in futures]
print(f"4 进程: {time.perf_counter() - start:.3f}s")
```

### 进程池和通信

```python-run
import multiprocessing

def square(x: int) -> int:
    return x * x

# Pool: 最常用的进程池接口
with multiprocessing.Pool(processes=4) as pool:
    results = pool.map(square, range(10))
print(f"pool.map: {results}")
```

### 多进程不共享内存

```python-run
from multiprocessing import Process

global_var = 0

def child() -> None:
    global global_var
    global_var = 42
    print(f"[子进程] global_var = {global_var}")

p = Process(target=child)
p.start(); p.join()
print(f"[主进程] global_var = {global_var}")  # 还是 0！

# 每个进程有独立的内存空间。global_var 在子进程中的修改不影响主进程。
# 共享数据必须用 multiprocessing.Value / Array / Manager
```

### 共享内存方式

| 方式 | 适用场景 | 性能 |
|------|---------|------|
| `multiprocessing.Value/Array` | 简单数值、数组 | 高（共享内存 + Lock） |
| `multiprocessing.Manager` | 复杂数据结构 | 低（通过代理进程通信） |
| `multiprocessing.Queue/Pipe` | 消息传递 | 中（序列化开销） |

> **多进程的核心成本**：启动进程（慢）、数据序列化（`pickle`）、进程间通信。不要为轻量任务开进程——开销可能超过收益。

### threading vs multiprocessing 选择

| 条件 | 推荐 |
|------|------|
| I/O 密集 + 需要共享内存 | `threading` |
| CPU 密集 | `multiprocessing` |
| 调用释放 GIL 的 C 库 | `threading` |
| 需要进程隔离（子进程崩溃不影响主进程） | `multiprocessing` |
| 大量短期任务 | 都不要——用 `concurrent.futures` 池 |

> 📖 官方文档：[multiprocessing](https://docs.python.org/3/library/multiprocessing.html)

## Part D: `concurrent.futures` —— 统一接口

`concurrent.futures` 提供了**统一的线程池/进程池 API**，让你用同一套接口切换底层并发模型。

### ThreadPoolExecutor vs ProcessPoolExecutor

```python-run
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

def task(x: int) -> int:
    return x * x

# 切换并发模型只需要改 Executor 的类型
with ThreadPoolExecutor(max_workers=4) as executor:
    results = executor.map(task, range(10))
print(f"线程池: {list(results)}")

# 换成一行即可切换到进程池
# with ProcessPoolExecutor(max_workers=4) as executor:
#     results = executor.map(task, range(10))
```

### map() vs submit()

```python-run
from concurrent.futures import ThreadPoolExecutor, as_completed
import time

def fetch(url: str) -> str:
    time.sleep({"a": 0.3, "b": 0.1, "c": 0.5}.get(url, 0.1))
    return f"OK({url})"

# map(): 保持输入顺序，简单直观 —— 适合"等我全部完成"
with ThreadPoolExecutor(max_workers=3) as executor:
    results = executor.map(fetch, ["a", "b", "c"])
    # 结果按输入顺序返回

# submit() + as_completed(): 按完成顺序处理 —— 适合"哪个先好先处理哪个"
with ThreadPoolExecutor(max_workers=3) as executor:
    futures = {executor.submit(fetch, url): url for url in ["a", "b", "c"]}
    for future in as_completed(futures):
        url = futures[future]
        print(f"最先完成: {url} → {future.result()}")
```

> **选择规则**：如果任务执行时间差异不大，用 `map()`（代码更简洁）。如果任务耗时差异很大，用 `submit()` + `as_completed()`（先完成先处理，减少总等待时间）。

### 实战：并行下载 + 处理

```python-run
from concurrent.futures import ThreadPoolExecutor, as_completed
import time, random

def download(url: str) -> str:
    time.sleep(random.uniform(0.05, 0.2))  # 模拟下载
    return f"<content of {url}>"

def process(content: str) -> int:
    return len(content)

urls = [f"https://api.example.com/item/{i}" for i in range(8)]

# 第一阶段：并行下载 → 第二阶段：逐个处理
with ThreadPoolExecutor(max_workers=4) as executor:
    download_futures = {executor.submit(download, url): url for url in urls}
    results: list[int] = []
    for future in as_completed(download_futures):
        results.append(process(future.result()))

print(f"下载了 {len(urls)} 个 URL，总处理大小: {sum(results)}")
```

> 📖 官方文档：[concurrent.futures](https://docs.python.org/3/library/concurrent.futures.html)

## Part E: 决策指南

```
你的任务是？
├── I/O 密集型（网络、文件、数据库）
│   ├── 需要共享内存状态 → threading
│   ├── 用 async/await 的代码库 → asyncio（下一章）
│   └── 简单并行 + 统一接口 → concurrent.futures.ThreadPoolExecutor
│
├── CPU 密集型（计算、数据处理）
│   ├── 简单并行 → concurrent.futures.ProcessPoolExecutor
│   ├── 需要精细控制 → multiprocessing
│   └── 调用释放 GIL 的 C 库 → threading（C 库内部释放 GIL 时真正并行）
│
└── 混合型
    ├── async + CPU → asyncio + ProcessPoolExecutor（asyncio.to_process? 暂无）
    └── threading + CPU → threading + ProcessPoolExecutor 混合
```

### 几条铁律

1. **不要为 CPU 密集型任务开线程**——你会更慢
2. **不要为每个请求开一个新进程**——用池（Pool / Executor）
3. **I/O 密集型：默认用 `ThreadPoolExecutor`**，代码最简洁
4. **CPU 密集型：默认用 `ProcessPoolExecutor`**，绕过 GIL
5. **需要精细控制的并发：去看下一章的 asyncio**

## 本章小结

- **GIL 本质**：GIL 是 CPython 的互斥锁，保证同一时刻只有一个线程执行 Python 字节码，其根源是引用计数内存管理的线程安全需求；I/O 等待时 GIL 被释放。
- **CPU vs I/O**：CPU 密集型多线程因 GIL 竞争反而更慢，必须用 `multiprocessing` 绕过；I/O 密集型多线程能有效并发，因为 I/O 等待期间释放 GIL。
- **线程安全**：GIL 仅保护单个字节码操作，`counter += 1` 等多步逻辑仍需 `Lock` 保护；`queue.Queue` 是线程间通信的标准方案，内置线程安全。
- **统一接口**：`concurrent.futures` 提供 `ThreadPoolExecutor` 和 `ProcessPoolExecutor` 的统一 API，`map()` 保序返回，`submit()` + `as_completed()` 按完成顺序处理。
- **决策铁律**：I/O 密集默认用 `ThreadPoolExecutor`，CPU 密集默认用 `ProcessPoolExecutor`，需要精细控制或高并发连接时用 asyncio。
