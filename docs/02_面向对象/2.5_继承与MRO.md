# 2.5 继承与 MRO

本章回答三个问题：Python 的继承机制**如何运作**、`super()` **真正做了什么**、多重继承**何时该用、何时该躲**。

## 5.1 单继承：不只是"照抄父类"

先看最简单的继承——子类从父类获得属性和方法：

```python-run
class Animal:
    def speak(self) -> str:
        return "..."

class Dog(Animal):
    def speak(self) -> str:          # 覆盖（Override）
        return "Woof!"

class Cat(Animal):
    pass                             # 完全继承

print(Dog().speak())  # Woof!
print(Cat().speak())  # ...        ← 来自 Animal
```

Python 在类上查找属性的路径是确定的：**先搜自己，再搜父类，再搜父类的父类**。这条路径就是"方法解析顺序"（Method Resolution Order，简称 MRO）。

用 `.mro()` 或 `.__mro__` 可以直观地看到它：

```python-run
class A: pass
class B(A): pass
class C(B): pass

print(C.__mro__)
# (C, B, A, object)  ← 查找顺序：先去 C，再去 B，再去 A，最后去 object
```

> Python 3 中**所有类都是新式类**（new-style class），最终都继承自 `object`。

> 📖 官方文档：[Inheritance](https://docs.python.org/3/tutorial/classes.html#inheritance), [MRO](https://www.python.org/download/releases/2.3/mro/)

## 5.2 C3 线性化：一条公式统治所有查找

单继承的查找顺序是明摆着的——一条直线往上走。但多重继承呢？如果有两个父类，该先搜谁？

Python 2.3 之前用的是"深度优先、从左到右"（旧式类 MRO），在很多场景下会产生反直觉的顺序。Python 2.3 引入的 **C3 线性化算法** 彻底解决了这个问题。

### C3 算法的核心规则

C3 保证三条性质：

1. **子类优先于父类**（monotonicity）：如果 A 是 B 的父类，A 永远排在 B 之前
2. **父类声明顺序保持**（local precedence）：`class D(A, B)` 中 A 的查找优先级高于 B
3. **所有父类的 MRO 保持**（consistency）：父类自己的 MRO 顺序不被破坏

C3 的合并公式（简化版）：

```
L[C(B1, B2, ..., Bn)] = C + merge(L[B1], L[B2], ..., L[Bn], [B1, B2, ..., Bn])
```

其中 `merge` 的规则是：从每个列表的**头部**找一个**不出现在任何其他列表尾部**的类，把它取出来放到结果中；重复直到所有列表为空。如果找不到这样的类，说明继承图无法线性化——Python 会直接报 `TypeError`。

### 手动演示一个 C3 计算

```
继承关系：
      A(最终父类)
     / \
    B   C
     \ /
      D(B, C)
```

已知 `object` 是所有类的源头，`L[object] = [object]`。

```
# 第一步：列出所有父类的 MRO
L[A]    = [A, object]
L[B]    = [B, A, object]     ← B 继承 A
L[C]    = [C, A, object]     ← C 继承 A

# 第二步：用 C3 公式计算 L[D]
L[D] = [D] + merge(L[B], L[C], [B, C])
     = [D] + merge([B, A, object], [C, A, object], [B, C])

# merge 过程：
# 取 B：B 不在 [C, A, object] 和 [B, C] 的尾部 → ✓ 取出 B
# → [D, B] + merge([A, object], [C, A, object], [C])
# 取 A：A 在 [C, A, object] 的尾部！→ ✗ 跳过
# 取 C：C 不在 [A, object] 和 [C] 的尾部 → ✓ 取出 C
# → [D, B, C] + merge([A, object], [A, object], [])
# 取 A：A 不在 [A, object] 的尾部 → ✓ 取出 A
# → [D, B, C, A] + merge([object], [object], [])
# 取 object：→ [D, B, C, A, object]

L[D] = [D, B, C, A, object]
```

验算——这正是 Python 实际给出的结果：

```python-run
class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass

print(D.__mro__)
# (<class '__main__.D'>, <class '__main__.B'>, <class '__main__.C'>,
#  <class '__main__.A'>, <class 'object'>)
```

### 什么时候 C3 会失败

不是所有继承图都能被线性化。当两个父类的 MRO 冲突且无法调和时，Python 会拒绝创建这个类：

```python-run
class X: pass
class Y: pass

class A(X, Y): pass             # OK：A 的 MRO = A, X, Y, object
class B(Y, X): pass             # OK：B 的 MRO = B, Y, X, object

try:
    class Impossible(A, B):     # 继承两个 MRO 冲突的类！
        pass
except TypeError as e:
    print(f"TypeError: {e}")
    # 错误信息大致: Cannot create a consistent method resolution
    # order (MRO) for bases X, Y
```

冲突点在于：A 说 "X 在 Y 之前"，B 说 "Y 在 X 之前"——C3 无法调和这个矛盾，于是直接拒绝。这不是 bug，是 Python 在**保护你不写出行为不可预测的代码**。

> 📖 官方文档：[C3 Method Resolution Order](https://www.python.org/download/releases/2.3/mro/)

## 5.3 super()：不是"父类"，是"MRO 里的下一个"

这是 Python 的 `super()` 最容易让人误解的地方。

在 Java / C++ 里，`super` 直接指向**父类**。但在 Python 里，`super()` 返回一个 **proxy 对象**，它做的事情是：**沿着 MRO 链，从当前类开始，找下一个实现了该方法的类**。

```python-run
class A:
    def who(self):
        return "A"

class B(A):
    def who(self):
        return "B → " + super().who()   # ← super() 去 MRO 里下一个

class C(A):
    def who(self):
        return "C → " + super().who()

class D(B, C):            # MRO: D → B → C → A → object
    def who(self):
        return "D → " + super().who()

print(D().who())
# D → B → C → A
#      ^^^^
# super() 在 B 里跳到了 C，不是 A！
# 因为 MRO 是 D, B, C, A——B 之后是 C，不是 A。
```

**关键认知**：`super()` 在**单继承** 中表现得像"调用父类"，这只是碰巧——单继承时 MRO 的下一个正好是父类。在**多重继承**中，`super()` 的"下一个"很可能不是直接父类。

> 📖 官方文档：[super()](https://docs.python.org/3/library/functions.html#super), [PEP 3135 – New Super](https://peps.python.org/pep-3135/)

## 5.4 协作式多重继承：钻石问题与解题思路

### 钻石问题是什么

```
      Root
       ↑
   ┌───┴───┐
  Left    Right
   └───┬───┘
       ↑
     Bottom
```

```python-run
class Root:
    def __init__(self):
        print("Root.__init__")
        self.name = "root"

class Left(Root):
    def __init__(self):
        print("Left.__init__")
        super().__init__()         # ← 沿着 MRO 走！
        self.left_only = True

class Right(Root):
    def __init__(self):
        print("Right.__init__")
        super().__init__()         # ← 沿着 MRO 走！
        self.right_only = True

class Bottom(Left, Right):
    def __init__(self):
        print("Bottom.__init__")
        super().__init__()         # ← 一调用，整条链串联起来

print("MRO:", [c.__name__ for c in Bottom.__mro__])
# MRO: ['Bottom', 'Left', 'Right', 'Root', 'object']

print("--- 创建 Bottom() ---")
b = Bottom()
print(f"name={b.name}, left_only={b.left_only}, right_only={b.right_only}")

# 输出：
# MRO: Bottom, Left, Right, Root, object
# --- 创建 Bottom() ---
# Bottom.__init__
# Left.__init__
# Right.__init__
# Root.__init__
# name=root, left_only=True, right_only=True
```

**如果不用 `super()`**，而是写 `Root.__init__(self)`，会发生什么？

```python-run
class LeftBad(Root):
    def __init__(self):
        print("LeftBad.__init__")
        Root.__init__(self)        # ✗ 硬编码调 Root，断了 MRO 链
        self.left_only = True

class RightBad(Root):
    def __init__(self):
        print("RightBad.__init__")
        Root.__init__(self)        # ✗ 同样硬编码
        self.right_only = True

class BottomBad(LeftBad, RightBad):
    def __init__(self):
        print("BottomBad.__init__")
        LeftBad.__init__(self)
        RightBad.__init__(self)    # ← 手动调两次，Root.__init__ 被执行两遍！

print("--- 不走 super() 的悲剧 ---")
BottomBad()
# BottomBad.__init__
# LeftBad.__init__
# Root.__init__        ← Root 第一次调用
# RightBad.__init__
# Root.__init__        ← Root 第二次调用！数据可能被覆盖，逻辑重复执行
```

**协作式多重继承的黄金法则**：

1. 每条链上的每个类都**必须调用 `super()`**
2. **参数签名要兼容**——不知道 `super()` 的下一个是谁，所以用 `*args, **kwargs` 传递
3. 在 `__init__` 的参数处理上尤其小心

```python-run
class Root:
    def __init__(self, **kwargs):
        print("Root.__init__")

class Left(Root):
    def __init__(self, left_arg=None, **kwargs):  # 吃掉自己的参数
        print(f"Left.__init__(left_arg={left_arg})")
        super().__init__(**kwargs)                # 余下的往下传

class Right(Root):
    def __init__(self, right_arg=None, **kwargs): # 吃掉自己的参数
        print(f"Right.__init__(right_arg={right_arg})")
        super().__init__(**kwargs)               # 余下的往下传

class Bottom(Left, Right):
    def __init__(self, **kwargs):
        print("Bottom.__init__")
        super().__init__(**kwargs)               # 全量往下传

Bottom(left_arg=1, right_arg=2)
# Bottom.__init__
# Left.__init__(left_arg=1)
# Right.__init__(right_arg=2)
# Root.__init__
```

> 📖 官方文档：[super() and cooperative multiple inheritance](https://docs.python.org/3/tutorial/classes.html#multiple-inheritance)

## 5.5 Mixin 模式：多重继承的正确打开方式

多重继承最健康的使用方式不是"一个东西同时是两种东西"（那叫组合更好），而是 **Mixin（混入类）**：小而专注的类，给主类增加一项**单一能力**。

### Mixin 的三个特征

1. **功能单一**：一个 Mixin 只做一件事（日志、序列化、缓存...）
2. **不实例化**：Mixin 本身不创造对象，只用来被继承
3. **放在父类列表最前面**：`class MyClass(LoggingMixin, SomeBase)`——确保 Mixin 的方法优先

```python-run
import time
import json

# ── Mixin 1：记录方法调用 ──
class LoggingMixin:
    """给任意类加上方法调用日志"""
    def _log(self, method_name, *args, **kwargs):
        print(f"[{time.strftime('%H:%M:%S')}] {self.__class__.__name__}."
              f"{method_name}({args}, {kwargs})")

# ── Mixin 2：序列化能力 ──
class SerializableMixin:
    """给任意类加上 to_json / from_json"""
    def to_json(self) -> str:
        data = {k: v for k, v in self.__dict__.items()
                if not k.startswith('_')}
        return json.dumps(data, ensure_ascii=False, default=str)

    @classmethod
    def from_json(cls, json_str: str):
        data = json.loads(json_str)
        return cls(**data)

# ── 业务类：组合 Mixin ──
class User(LoggingMixin, SerializableMixin):
    def __init__(self, name: str, age: int):
        self.name = name
        self.age = age

    def greet(self):
        self._log("greet")
        return f"Hello, I'm {self.name}"

u = User("Alice", 30)
print(u.greet())             # 带日志
print(u.to_json())           # 序列化
```

Mixin 和 C3 MRO 配合得很好：因为 Mixin 通常没有自己的父类（或只继承 `object`），它们插入 MRO 链时不会制造冲突，只是"延长"了查找链。

### Mixin 常见模式速查

| Mixin 类型 | 示例能力 | 典型方法 |
|-----------|---------|---------|
| 日志 | 记录方法调用/状态变更 | `_log()`, `log_state()` |
| 序列化 | JSON/XML 转换 | `to_json()`, `from_json()` |
| 比较 | 自定义排序 | `__lt__`, `__eq__`, `__hash__` |
| 上下文管理 | 自动资源管理 | `__enter__`, `__exit__` |
| 不可变 | 冻结对象修改 | `__setattr__`, `__delattr__` |

> 📖 **延伸**：Django 的类视图大量使用 Mixin（`LoginRequiredMixin`、`UserPassesTestMixin` 等），这是多重继承在生产代码中被最广泛使用的模式之一。

## 5.6 __init__ 和 super()：为什么单继承也该用

很多人觉得："我只有一个父类，直接 `Parent.__init__(self, ...)` 省事。" 这在 Python 中**是坏习惯**。

```python-run
class Engine:
    def __init__(self, horsepower: int):
        self.horsepower = horsepower
        print(f"Engine: {self.horsepower}hp")

# ── 错误写法 ──
class CarBad(Engine):
    def __init__(self, horsepower: int, seats: int):
        Engine.__init__(self, horsepower)  # ✗ 硬编码
        self.seats = seats

# ── 正确写法 ──
class CarGood(Engine):
    def __init__(self, horsepower: int, seats: int):
        super().__init__(horsepower)       # ✓ 用 super()
        self.seats = seats
```

**为什么单继承也要用 `super()`？**

1. **重构安全**：将来如果 `Car` 增加了第二个父类，或 `Engine` 的继承链变了，`super()` 自动适应——硬编码的 `Engine.__init__` 不会
2. **Mixin 友好**：即使你现在只继承一个类，将来有人可能用 Mixin 扩展你的类。如果不用 `super()`，Mixin 的 `__init__` 不会被执行
3. **代码意图清晰**：`super()` 说的是"按照 MRO 链调用下一个"，而 `Engine.__init__` 说的是"就调 Engine 这个类"——前者是合作，后者是独裁

## 5.7 issubclass() 与 isinstance()

```python-run
class Animal: pass
class Dog(Animal): pass
class Robot: pass

# isinstance：检查实例
d = Dog()
print(isinstance(d, Dog))      # True
print(isinstance(d, Animal))   # True  ← 子类实例也是父类实例
print(isinstance(d, Robot))    # False
print(isinstance(d, object))   # True  ← 万物皆 object

# issubclass：检查类之间的关系
print(issubclass(Dog, Animal))  # True
print(issubclass(Dog, Dog))     # True  ← 自己是自己的子类！
print(issubclass(Dog, object))  # True

# 检查多个类型
print(isinstance(d, (Animal, Robot)))     # True（满足其一）
print(issubclass(Dog, (Animal, Robot)))   # True
```

**性能提示**：`isinstance()` 在 Python 中是高度优化的 O(1) 操作——不是遍历 MRO 链。CPython 内部使用 `__class__` 指针和类型缓存（type version tag）来加速判断。

> 📖 官方文档：[isinstance()](https://docs.python.org/3/library/functions.html#isinstance), [issubclass()](https://docs.python.org/3/library/functions.html#issubclass)

## 5.8 抽象基类（ABC）：定义"必须做什么"的契约

抽象基类（Abstract Base Class，简称 ABC）是一种**不能被实例化**的类，它定义了子类**必须实现**的方法。

```python-run
from abc import ABC, abstractmethod

class Animal(ABC):
    @abstractmethod
    def speak(self) -> str:
        """子类必须实现这个方法"""
        ...                                     # 方法体可以为空（... 或 pass）

# animal = Animal()  # TypeError: Can't instantiate abstract class Animal
                      # with abstract method speak

class Dog(Animal):
    def speak(self) -> str:
        return "Woof!"

class Cat(Animal):
    pass        # ← 没有实现 speak()，Cat 也是抽象的

dog = Dog()
print(dog.speak())  # Woof!

# cat = Cat()       # TypeError: Can't instantiate abstract class Cat
                    # with abstract method speak
```

### @abstractmethod 的细节

```python-run
from abc import ABC, abstractmethod

class DataProcessor(ABC):
    @abstractmethod
    def process(self, data: list) -> list:
        """处理数据——子类必须实现"""
        ...

    @abstractmethod
    def validate(self, data: list) -> bool:
        """校验数据——子类必须实现"""
        ...

    def run(self, data: list) -> list:
        """模板方法：定义算法骨架"""
        if not self.validate(data):
            raise ValueError("数据校验失败")
        return self.process(data)        # ← 依赖抽象方法，但不实现它们
```

这里用了**模板方法模式（Template Method Pattern）**：ABC 中的具体方法（`run()`）定义算法骨架，**调用**抽象方法但不实现它们；子类只需要填上抽象方法，算法结构由 ABC 控制。

### ABC.register()：虚拟子类

不想继承 ABC，但想让 `isinstance` 和 `issubclass` 认账？用 `register()`：

```python-run
from abc import ABC, abstractmethod

class Flyable(ABC):
    @abstractmethod
    def fly(self) -> str: ...

class Bird:
    def fly(self) -> str:
        return "飞起来了"

Flyable.register(Bird)           # ← 注册为虚拟子类，不需要继承

print(issubclass(Bird, Flyable)) # True
print(isinstance(Bird(), Flyable))  # True   ← 但 Bird 不继承 Flyable！
```

`register()` 的典型场景：

- **外部库的类**：你不能修改其源代码，要它继承你的 ABC 不现实
- **collections.abc**：Python 内置的 ABC（如 `Iterable`、`Sequence`、`Mapping`）大量使用 `register()` 来让内置类型"属于"抽象类

```python-run
from collections.abc import Sequence

# list 和 str 被注册为 Sequence 的虚拟子类
print(issubclass(list, Sequence))  # True
print(issubclass(str, Sequence))   # True

# 但你自己定义的类需要"实现协议"才能真正用：
class MyList:
    def __getitem__(self, idx): ...  # 只有这个还不够...
    def __len__(self): ...

Sequence.register(MyList)
print(issubclass(MyList, Sequence))  # True ← 注册了就算，不检查你是否真实现了！
```

**注意**：`register()` 不做运行时检查——你注册了一个没有实现 `fly()` 的类，Python 不会阻止你。`isinstance` 和 `issubclass` 会返回 `True`，但调用 `fly()` 时一样会报错。

> 📖 官方文档：[abc — Abstract Base Classes](https://docs.python.org/3/library/abc.html), [collections.abc](https://docs.python.org/3/library/collections.abc.html)

## 5.9 何时不用多重继承

多重继承很强大，但也很容易写出难以理解的代码。以下信号提醒你该考虑**不用**多重继承：

| 信号 | 替代方案 |
|------|---------|
| 两个父类的职责有重叠 | **组合**：`self.engine = Engine()` |
| 父类的数量超过 3 个 | **组合 + 单一主继承** |
| 父类之间有同名方法且语义不同 | **重新设计**——这是 C3 也救不了的架构问题 |
| 你在纠结 `self` 到底该传给谁 | **不要用多重继承** |
| 你需要看 MRO 才知道方法会从哪来 | 你的读者也得看，这是认知负担 |

**Python 的设计哲学**本身就是"简单优于复杂"（`import this` 的第二句）。Guido 对多重继承的态度很明确：

> "It is possible to misuse multiple inheritance, and this is one of the reasons why some languages (Java, for example) take a different approach. But when used appropriately, it is a powerful and elegant mechanism." — Guido van Rossum

翻译：多重继承可能被滥用，Java 因此选择了完全不同的方向。但当恰当地使用时，它是一种强大而优雅的机制。

**经验法则**：
- 多重继承的**唯一健康模式**是：一个主继承链 + 若干 Mixin
- 两个"平等"的父类同时继承（`class Car(Engine, Chassis)`）→ 几乎总是应该用组合

## 本章小结

- C3 线性化算法保证子类优先于父类、父类声明顺序保持、父类 MRO 不被破坏；无法线性化时 Python 直接报 `TypeError` 拒绝创建类。
- `super()` 不是"父类"，而是"沿 MRO 链的下一个类"——在单继承中恰好是父类，在多重继承中完全可能跳到非直接父类。
- 协作式多重继承的黄金法则：每条链上的每个类都必须调用 `super()`，参数用 `**kwargs` 传递以保持签名兼容。
- Mixin 是多重继承的正确用法——功能单一、不单独实例化、放在父类列表前面；两个"平等"父类同时继承几乎总是应该用组合替代。
- `ABC` 定义子类必须实现的方法契约；`register()` 可注册虚拟子类但不检查实现；`isinstance()`/`issubclass()` 是高度优化的 O(1) 操作。

## 思考题

1. `class D(B, C)` 中 B 和 C 都继承了 A。在没有 `super()` 的情况下，`D.__init__` 中手动调用 `B.__init__` 和 `C.__init__` 会导致 `A.__init__` 被执行几次？用代码验证你的答案。

2. `super()` 在类方法（`@classmethod`）中也能使用吗？如果 `super().method()` 解析到的类没有定义该方法，会出现什么情况？写代码验证。

3. 你有一个库提供了 `UserView(LoginRequiredMixin, TemplateView)`。阅读其 MRO，解释为什么 `LoginRequiredMixin` 要放在 `TemplateView` 前面。

4. `ABC.register()` 注册的"虚拟子类"不会真正继承任何实现。如果调用一个虚拟子类没有实现的抽象方法，会发生什么？这与直接继承 ABC 的区别在哪里？

5. 设计一个场景：你需要给三个不相关的类（`User`、`Product`、`Order`）增加"时间戳"能力（记录 `created_at` 和 `updated_at`）。你会用 Mixin 还是装饰器？还是别的方式？说明理由并写代码验证。
