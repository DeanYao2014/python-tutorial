# 2.8 元类实战

上一章（[2.1 类与元类的基础](2.1_类与元类的基础)）讲了元类（Metaclass）的原理——`type` 是类的类，`__new__` 和 `__init__` 控制类的创建过程。但原理懂了，什么时候**真的需要**元类？

本章回答三个问题：**`__init_subclass__` 能替代多少元类场景**、**真正需要元类时怎么写**、**什么时候绝对不该用元类**。

## 8.1 元类的"降级替代"：`__init_subclass__`

### 它是什么

Python 3.6（PEP 487）引入了 `__init_subclass__`——当一个类被**继承**时，父类的 `__init_subclass__` 自动被调用。它和元类的区别在于**切入点不同**：

| | `__init_subclass__` | 元类 `__new__` / `__init__` |
|---|---|---|
| 触发时机 | 子类定义完成**之后** | 类创建**过程中** |
| 能做什么 | 注册子类、检查约束、修改子类属性 | 修改类命名空间 **before** 类被创建 |
| 复杂度 | 低——只是一个类方法 | 高——需要理解 `type` 的调用协议 |
| 适用人群 | 99% 的人 | 1% 的人 |

> 元类的 `__init__` 在类体执行完、命名空间收集好之后才调用——这一点和 `__init_subclass__` 是同一个阶段。真正独有的是元类 `__new__`：它能拿到**原始的** `namespace` dict，在类被创建**之前**修改它。

> 📖 官方文档：[__init_subclass__](https://docs.python.org/3/reference/datamodel.html#customizing-class-creation), [PEP 487](https://peps.python.org/pep-0487/)

### 场景一：插件自动注册

你需要一个插件系统：任何人写一个 `Parser` 子类，它自动出现在可用解析器列表中。

```python-run
class Plugin:
    """插件基类——子类定义时自动注册到 _plugins"""
    _plugins: dict[str, type["Plugin"]] = {}

    def __init_subclass__(cls, **kwargs) -> None:
        super().__init_subclass__(**kwargs)
        name = getattr(cls, "plugin_name", cls.__name__)
        if name in cls._plugins:
            raise TypeError(f"插件名 '{name}' 已被注册")
        cls._plugins[name] = cls

    @classmethod
    def get(cls, name: str) -> type["Plugin"] | None:
        return cls._plugins.get(name)


class TextParser(Plugin):
    plugin_name = "text"

    def parse(self, data: str) -> list[str]:
        return data.split()


class JsonParser(Plugin):
    plugin_name = "json"

    def parse(self, data: str) -> list[str]:
        import json
        return list(json.loads(data).keys())


# 无需手动注册——子类定义即注册
print(list(Plugin._plugins.keys()))  # ['TextParser', 'JsonParser']
parser = Plugin.get("json")()
print(parser.parse('{"name": "Alice"}'))  # ['name']
```

**核心价值**：你不需要维护一个 `ALL_PARSERS = [TextParser, JsonParser]` 的列表——谁写新解析器，谁就自动在了。

> 类似模式广泛应用于 pytest 插件、Flask 扩展、Django Model 等框架。

### 场景二：强制子类约束

```python-run
class Animal:
    required_attrs: tuple[str, ...] = ("sound",)

    def __init_subclass__(cls, **kwargs) -> None:
        super().__init_subclass__(**kwargs)
        for attr in cls.required_attrs:
            if not hasattr(cls, attr) or getattr(cls, attr) is None:
                raise TypeError(
                    f"{cls.__name__} 必须定义类属性 '{attr}'"
                )


class Dog(Animal):
    sound = "汪汪"


# class Cat(Animal):   # TypeError: Cat 必须定义类属性 'sound'
#     pass

dog = Dog()
print(dog.sound)  # 汪汪
```

这和 ABC 的 `@abstractmethod` 不同——`@abstractmethod` 检查的是方法是否覆盖，而我们这里检查的是**类属性**是否存在。两者互补。

### `__init_subclass__` 够用吗？

以下场景 **`__init_subclass__` 足够**：

- 子类注册（插件系统）
- 子类约束检查（必须定义某属性/方法）
- 给子类动态添加属性或方法
- 修改子类的某些元信息（如 `__repr__` 的生成逻辑）

以下场景 **需要元类**：

- 修改类创建前的**原始命名空间**（如自动把 `snake_case` 属性名变成 `camelCase`）
- 控制类本身的创建过程（如禁止创建有某些特征的类——在类体执行完之前就拦截）
- 需要**跨继承层级**的全局行为（如所有类的注册表必须是一个全局单例）

> **经验法则**：如果你发现自己写 `class Foo(SomeBase, metaclass=MyMeta)`，先问：能不能用 `__init_subclass__` 替代 `MyMeta`？九成的情况可以。

## 8.2 `__set_name__`：描述符的"自我认知"

### 它是什么

描述符（Descriptor）有一个经典痛点：描述符对象不知道自己在所属类中的属性名。

```python
class Price:
    """一个'瞎'的描述符——不知道自己叫 price"""
    def __get__(self, obj, objtype=None):
        return obj._value  # 硬编码了 _value——但万一属性名是 amount 呢？

    def __set__(self, obj, value):
        if value < 0:
            raise ValueError("价格不能为负")  # 报错时也说不清是哪个字段
        obj._value = value
```

Python 3.6（PEP 487）引入 `__set_name__`：当类创建完成时，Python 自动调用每个描述符的 `__set_name__(self, owner, name)`，传入**所属的类**和**属性名**。

```python-run
class Validated:
    """带校验的描述符——通过 __set_name__ 自动获取字段名"""

    def __init__(self, expected_type: type):
        self.expected_type = expected_type
        self.name: str | None = None
        self.storage_name: str | None = None

    def __set_name__(self, owner: type, name: str) -> None:
        """类创建时自动调用——描述符终于知道自己的名字了"""
        self.name = name
        self.storage_name = f"_{name}"

    def __get__(self, obj, objtype=None):
        if obj is None:
            return self
        return getattr(obj, self.storage_name)

    def __set__(self, obj, value):
        if not isinstance(value, self.expected_type):
            raise TypeError(
                f"{self.name} 期望 {self.expected_type.__name__}，"
                f"实际是 {type(value).__name__}"
            )
        setattr(obj, self.storage_name, value)


class Person:
    name = Validated(str)
    age = Validated(int)

    def __init__(self, name: str, age: int):
        self.name = name
        self.age = age

    def __repr__(self):
        return f"Person(name={self.name!r}, age={self.age})"


p = Person("Alice", 30)
print(p)

try:
    p.age = "not a number"
except TypeError as e:
    print(f"校验生效: {e}")  # "age 期望 int，实际是 str"
```

不用 `__set_name__` 的话，你得写成 `age = Validated(int, "age")`——属性名写两遍，迟早不同步。

> 📖 官方文档：[__set_name__](https://docs.python.org/3/reference/datamodel.html#object.__set_name__), [PEP 487](https://peps.python.org/pep-0487/)

### `__set_name__` 的生态影响

这个机制让描述符在 Python 生态中真正实用了起来。以下库都依赖它：

- **Pydantic** — `Field()` 描述符通过 `__set_name__` 知道字段名
- **SQLAlchemy 2.0** — `mapped_column()` 自动映射列名
- **attrs / dataclasses** — 字段对象通过 `__set_name__` 建立反向引用

## 8.3 元类实战：三类真实场景

上面讲了"能用 `__init_subclass__` 就别用元类"。但如果你确实需要修改类**创建前**的命名空间，元类是唯一的答案。

### 场景一：简易 ORM —— 自动收集字段、生成表名

这是元类最经典的应用场景——Django、SQLAlchemy、Peewee 都在用。

```python-run
class ModelMeta(type):
    """ORM 元类：自动收集字段、生成表名"""

    def __new__(mcs, name, bases, namespace):
        # 收集当前类中所有非私有的类属性作为"字段"
        fields = {
            k: v for k, v in namespace.items()
            if not k.startswith('_') and not callable(v)
        }
        namespace['_fields'] = fields

        # 自动生成表名（类名小写 + s）
        if 'table_name' not in namespace:
            namespace['table_name'] = name.lower() + 's'

        return super().__new__(mcs, name, bases, namespace)


class Model(metaclass=ModelMeta):
    """ORM 基类"""
    _fields: dict = {}
    table_name: str = ""

    def save(self) -> None:
        columns = ', '.join(self._fields.keys())
        placeholders = ', '.join('?' for _ in self._fields)
        values = [getattr(self, f) for f in self._fields]
        sql = f"INSERT INTO {self.table_name} ({columns}) VALUES ({placeholders})"
        print(f"SQL: {sql}")
        print(f"参数: {values}")


class User(Model):
    name = ""
    age = 0
    email = ""


class Product(Model):
    title = ""
    price = 0.0


print(f"User 表名:  {User.table_name}")    # users
print(f"User 字段:  {User._fields}")        # {'name': '', 'age': 0, 'email': ''}
print(f"Product 表名: {Product.table_name}") # products
print(f"Product 字段: {Product._fields}")   # {'title': '', 'price': 0.0}

u = User()
u.name, u.age, u.email = "Bob", 25, "bob@test.com"
u.save()
```

**为什么这里不用 `__init_subclass__`？** 因为自动生成 `table_name` 需要用类名 `name` 做字符串运算，而在 `__init_subclass__` 中虽然能拿到 `cls.__name__`，但给**已创建的类**修改 `table_name` 类属性和在**创建过程中**把 `table_name` 放进 `namespace`，语义上后者更干净——属性从类诞生那一刻就存在。

> 📖 官方文档：[metaclasses](https://docs.python.org/3/reference/datamodel.html#metaclasses)

### 场景二：注册表模式 —— 类定义即注册

```python-run
class RegistryMeta(type):
    """元类注册表——类定义时自动注册到中央字典"""
    _registry: dict[str, type] = {}

    def __new__(mcs, name, bases, namespace):
        cls = super().__new__(mcs, name, bases, namespace)
        if bases:  # 只注册子类，不注册基类本身
            mcs._registry[name] = cls
        return cls


class Command(metaclass=RegistryMeta):
    """命令基类"""

    def execute(self) -> str:
        raise NotImplementedError


class SaveCommand(Command):
    def execute(self) -> str:
        return "保存文件"


class OpenCommand(Command):
    def execute(self) -> str:
        return "打开文件"


class DeleteCommand(Command):
    def execute(self) -> str:
        return "删除文件"


# 自动发现所有命令
print(list(RegistryMeta._registry.keys()))
# ['SaveCommand', 'OpenCommand', 'DeleteCommand']

for name, cmd_cls in RegistryMeta._registry.items():
    print(f"{name}: {cmd_cls().execute()}")
```

> 这和 `__init_subclass__` 的插件注册实现的功能一样。区别在于：元类版本的注册表挂在**元类**上（`RegistryMeta._registry`），`__init_subclass__` 版本的注册表挂在**基类**上（`Plugin._plugins`）。语义上的权衡：注册表是"类型的元信息"（用元类）还是"基类的类属性"（用 `__init_subclass__`）？多数场景下后者更直观。

### 场景三：Singleton —— 以及为什么你不需要它

```python-run
class SingletonMeta(type):
    _instances: dict[type, object] = {}

    def __call__(cls, *args, **kwargs):
        """类被'调用'（即实例化）时，返回已有的实例"""
        if cls not in cls._instances:
            cls._instances[cls] = super().__call__(*args, **kwargs)
        return cls._instances[cls]


class AppConfig(metaclass=SingletonMeta):
    def __init__(self) -> None:
        self.database_url: str = "sqlite:///:memory:"
        self.debug: bool = True


a = AppConfig()
b = AppConfig()
print(f"a is b: {a is b}")                # True
a.database_url = "postgresql://prod"
print(b.database_url)                      # "postgresql://prod" —— 同一个实例
```

**Singleton 元类很酷，但 Python 中几乎不需要**：

```python
# Python 方式：模块级单例 —— 零代码，零魔法
# config.py
database_url: str = "sqlite://"
debug: bool = True

# 其他文件
import config
config.database_url = "postgresql://prod"
```

Python 模块天然是单例——`import` 只会执行一次模块代码，第二次 `import` 直接返回 `sys.modules` 中的缓存。你不需要元类来实现 Singleton，**模块就是 Python 的 Singleton**。

> 真正需要 Singleton 元类的场景极少——通常只有当你需要**延迟初始化**或者需要**单例继承**时。如果你只是在找全局配置管理，用模块。

## 8.4 什么时候不该用元类

> "Metaclasses are deeper magic than 99% of users should ever worry about. If you wonder whether you need them, you don't."  
> —— **Tim Peters**, Python 之禅的作者

### 决策树

你需要**在类定义时**自动做一些事情。问自己：

```
问题 1: 能用一个 @类装饰器 完成吗？
  ├── 是 → 用类装饰器。够简单，够清晰。
  └── 否 → 问题 2

问题 2: 能在类创建之后（__init_subclass__）完成吗？
  ├── 是 → 用 __init_subclass__。覆盖 90% 场景。
  └── 否 → 问题 3

问题 3: 你必须修改类创建之前的原始命名空间吗？
  ├── 是 → 用元类。这是元类真正独有能力。
  └── 否 → 回到问题 2，你大概率漏了什么。
```

### 三个工具的对比

| 工具 | 触发时机 | 典型场景 |
|------|---------|---------|
| 类装饰器 `@decorator` | 类创建**之后**（等价于 `Foo = decorate(Foo)`） | 注册、包裹、添加属性 |
| `__init_subclass__` | 子类创建**之后** | 插件注册、子类约束、动态属性注入 |
| 元类 `__new__` | 类创建**之前** | 修改命名空间、跨层级全局行为 |

### "元类滥用"的典型症状

```python
# 反例：元类做了一件类装饰器就能做的事
class UnnecessaryMeta(type):
    def __new__(mcs, name, bases, namespace):
        cls = super().__new__(mcs, name, bases, namespace)
        cls.created_at = "2024-01-01"  # 只是加个属性而已！
        return cls

# 正解：一个 @ 的事
def add_created_at(cls):
    cls.created_at = "2024-01-01"
    return cls

@add_created_at
class MyClass:
    pass
```

**元类会"传染"**：如果 `A` 用了 `MetaA`，`B` 用了 `MetaB`，那么 `C` 的父类列表 `(A, B)` 中的元类必须兼容——Python 会用继承链中最具体的元类。这意味着一个项目中如果有人引入了元类，后续的继承者都被迫理解这个元类的行为。非必要不元类。

> 📖 官方文档：[Determining the appropriate metaclass](https://docs.python.org/3/reference/datamodel.html#determining-the-appropriate-metaclass)

## 本章小结

- `__init_subclass__` 覆盖 90% 的元类场景——插件自动注册、子类约束检查、动态属性注入都能用它完成，比元类简单得多。
- `__set_name__` 让描述符在类创建时自动获取字段名，彻底解决了属性名需要写两遍（`name = Validated(int, "name")`）的痛点。
- 元类 `__new__` 的独有能力是修改类创建**之前**的原始命名空间——如 ORM 自动收集字段、自动生成表名，这些无法用 `__init_subclass__` 做到。
- Python 模块天然是单例（`import` 只在 `sys.modules` 中执行一次），大多数 Singleton 场景根本不需要元类。
- 决策顺序：能用品装饰器就别用 `__init_subclass__`，能用 `__init_subclass__` 就别用元类——元类会"传染"，增加所有人的认知负担。
