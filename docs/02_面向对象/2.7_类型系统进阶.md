# 2.7 类型系统进阶

本章回答三个问题：Python 类型标注**能做什么**、**为什么还不够**、Pydantic 如何**让标注在运行时干活**。

## 7.1 为什么需要类型标注

Python 3.5（PEP 484）引入类型标注语法的第一天，Guido 就说得清楚——这是**标注**，不是约束。

```python
def greet(name: str) -> str:
    return "Hello, " + name

greet(42)        # Python 3.12 运行：TypeError: can only concatenate str
greet(["world"]) # 同样运行时报错——标注没起作用
```

类型标注的价值不在运行时，而在**编码时**：

| 维度 | 作用 |
|------|------|
| IDE 补全 | `.` 后自动列出可用属性 |
| 类型检查器 | mypy / pyright 在 CI 中阻止类型错误 |
| 文档 | 函数签名即文档，不需要注释描述参数类型 |
| 重构 | 改名/改类型时自动发现所有受影响的调用点 |

> 📖 官方文档：[typing — Support for type hints](https://docs.python.org/3/library/typing.html)

## 7.2 泛型三剑客

静态类型系统最核心的能力是**泛型**——让一个函数/类能处理多种类型，同时保持类型安全。

### TypeVar：单一类型的泛化

```python
from typing import TypeVar

T = TypeVar('T')

def first(items: list[T]) -> T:      # 输入什么类型的列表，返回什么类型
    return items[0]

n = first([1, 2, 3])           # mypy 推断: n 是 int
s = first(['a', 'b', 'c'])     # mypy 推断: s 是 str
```

`TypeVar` 可以加边界约束：

```python
from collections.abc import Sized

B = TypeVar('B', bound=Sized)  # B 必须是"有长度"的类型

def longest(a: B, b: B) -> B:
    return a if len(a) >= len(b) else b

longest([1, 2], [3, 4, 5])    # OK：list 有 __len__
longest(1, 2)                  # mypy 报错: int 没有 __len__
```

### ParamSpec（3.10+）：捕获函数签名

```python
from typing import ParamSpec, Callable

P = ParamSpec('P')

def log_call(func: Callable[P, T]) -> Callable[P, T]:
    """装饰器，原函数的参数签名完整保留"""
    def wrapper(*args: P.args, **kwargs: P.kwargs) -> T:
        print(f"调用 {func.__name__}({args}, {kwargs})")
        return func(*args, **kwargs)
    return wrapper

@log_call
def greet(name: str, *, greeting: str = "Hello") -> str:
    return f"{greeting}, {name}"
# wrapper 的签名和 greet 完全一致——IDE 能正确补全 name 和 greeting
```

> 不用 `ParamSpec` 时，装饰器会把参数签名变成 `(*args: Any, **kwargs: Any)`，IDE 就"瞎了"。

### TypeVarTuple（3.11+）：变长类型参数

```python
from typing import TypeVarTuple

Ts = TypeVarTuple('Ts')

def make_tuple(*args: *Ts) -> tuple[*Ts]:
    return args

result = make_tuple(1, "hi", 3.14)
# mypy 推断: tuple[int, str, float]  ← 精确到每个位置！
```

> 📖 官方文档：[TypeVar](https://docs.python.org/3/library/typing.html#typing.TypeVar), [ParamSpec](https://docs.python.org/3/library/typing.html#typing.ParamSpec), [TypeVarTuple](https://docs.python.org/3/library/typing.html#typing.TypeVarTuple)

## 7.3 Protocol：鸭子类型的静态表达

Python 的哲学是"如果它走起来像鸭子，叫起来像鸭子，那它就是鸭子"。但传统类型标注要求你显式声明关系——直到 Protocol 出现。

```python
from typing import Protocol

class Quackable(Protocol):
    def quack(self) -> str: ...

class Duck:
    def quack(self) -> str:
        return "嘎嘎"

class DuckPerson:
    def quack(self) -> str:
        return "我学鸭叫"

def make_quack(animal: Quackable) -> str:
    return animal.quack()

make_quack(Duck())       # OK——Duck 不继承 Quackable，但有 quack()
make_quack(DuckPerson()) # OK——mypy 看的是结构，不看继承链
```

**Protocol vs ABC：**

| | ABC（抽象基类） | Protocol |
|---|---|---|
| 匹配方式 | 显式继承或 `register()` | 结构匹配（有方法就算） |
| 侵入性 | 需修改目标类 | 零侵入 |
| 适用场景 | 定义框架内的行为契约 | 描述外部类型的"能力" |

> 📖 官方文档：[Protocols](https://docs.python.org/3/library/typing.html#typing.Protocol)

## 7.4 Self（3.11+）：让子类知道自己是谁

链式调用在 Python 中很常见，但以前要准确标注返回类型很痛苦：

```python
from typing import Self

class Shape:
    color: str = "black"

    def set_color(self, color: str) -> Self:  # ← 返回"自己的类型"
        self.color = color
        return self

class Circle(Shape):
    def set_radius(self, radius: float) -> Self:
        ...
        return self

# mypy 精确推断：Circle.set_color() 返回 Circle！
circle = Circle().set_color("red").set_radius(5.0)
```

3.11 之前得写 `TypeVar('T', bound='Shape')`——又绕又啰嗦。`Self` 直接表达了"返回自己的类型"这个简单语义。

> 📖 官方文档：[Self](https://docs.python.org/3/library/typing.html#typing.Self)

## 7.5 @overload：同一函数，多种签名

Python 不支持真正的函数重载，但可以用 `@overload` 精确描述多种调用方式：

```python
from typing import overload

@overload
def get(key: str) -> str | None: ...

@overload
def get(key: str, default: str) -> str: ...

def get(key: str, default: str | None = None) -> str | None:
    """实际实现——不写类型标注"""
    data = {"name": "Alice"}
    return data.get(key, default)

# mypy 处理：
name = get("name")          # 推断: str | None
name2 = get("name", "Bob")  # 推断: str（不会是 None！）
```

> 📖 官方文档：[@overload](https://docs.python.org/3/library/typing.html#typing.overload)

## 7.6 约束型类型

### Literal：限定取值范围

```python
from typing import Literal

def align(text: str, *, side: Literal["left", "right", "center"]) -> str:
    ...
# IDE 自动补全 "left" | "right" | "center"，不会写错
```

### Final：防止意外覆盖

```python
from typing import Final

API_URL: Final = "https://api.example.com"
# API_URL = "其他"  # mypy 报错

class Base:
    TIMEOUT: Final[int] = 30
# class Sub(Base):
#     TIMEOUT = 60     # mypy 报错：不能覆盖 Final 属性
```

### TypedDict：描述字典的"形状"

```python
from typing import TypedDict, NotRequired  # NotRequired: 3.11+

class User(TypedDict):
    name: str
    age: int
    email: NotRequired[str]   # 可选字段

alice: User = {"name": "Alice", "age": 30}
# bob: User = {"name": "Bob"}    # mypy 报错：缺少 age
```

**TypedDict vs dataclass 的选择：**

| 场景 | 用什么 |
|------|--------|
| 外部 JSON / API 响应 / 数据库查询结果 | TypedDict |
| 内部业务对象，需要方法、验证、可变状态 | dataclass |
| 需要运行时校验 | Pydantic BaseModel |

> 📖 官方文档：[Literal](https://docs.python.org/3/library/typing.html#typing.Literal), [Final](https://docs.python.org/3/library/typing.html#typing.Final), [TypedDict](https://docs.python.org/3/library/typing.html#typing.TypedDict)

## 7.7 类型只是标注——这是好事

```python
import typing
import dataclasses

def check(x: int) -> str:
    return str(x)

# Python 运行时的态度：完全不看你的标注
check("hello")          # 运行成功，返回 "hello"
check([1, 2, 3])        # 运行成功，返回 "[1, 2, 3]"

# 要拿到标注，需要主动去取：
print(typing.get_type_hints(check))  # {'x': int, 'return': str}
```

`from __future__ import annotations`（PEP 563，3.7-3.10 默认需要，3.11+ 可选）会把标注存为字符串而非立即求值，解决了循环引用问题。PEP 649（3.14+）正在进一步改写这套机制。

**类型标注不运行，是刻意的设计**——它保证了：

- 引入 typing 不会让运行时代码变慢
- 类型标注可以是复杂表达式（如 `list[dict[str, int | None]]`）而不会在 import 时求值崩溃

但这意味着：如果你的代码依赖"别人传入正确的类型"，类型标注帮不了你——你需要运行时校验。

> 📖 官方文档：[get_type_hints](https://docs.python.org/3/library/typing.html#typing.get_type_hints), [PEP 649](https://peps.python.org/pep-0649/)

## 7.8 Pydantic：让标注在运行时干活

Pydantic 是 Python 生态最成功的数据校验库之一。它的核心理念：**把类型标注当作运行时校验规则**。

```python
from pydantic import BaseModel

class User(BaseModel):
    name: str         # str → 运行时段名必须是 str
    age: int          # int → 自动转换 "25" 为 25
    email: str

# 正常创建
alice = User(name="Alice", age=30, email="alice@example.com")

# 自动类型转换
bob = User(name="Bob", age="25", email="bob@example.com")
print(type(bob.age))  # <class 'int'> —— "25" 被转成了 25

# 校验失败——抛出 ValidationError
try:
    User(name="Eve", age="not a number", email="eve@example.com")
except Exception as e:
    print(e)
```

### 校验链路

Pydantic 的处理顺序是一个清晰的流水线：

```
原始数据 → 类型强制转换 → field_validator → model_validator → 最终对象
```

```python
from pydantic import BaseModel, field_validator, model_validator
from typing import Self

class User(BaseModel):
    name: str
    age: int
    email: str

    @field_validator('age')          # 单个字段
    @classmethod
    def check_age(cls, v: int) -> int:
        if v < 0 or v > 150:
            raise ValueError('年龄不合理')
        return v

    @model_validator(mode='after')   # 所有字段就绪后
    def check_email(self) -> Self:
        if '@' not in self.email:
            raise ValueError('邮箱格式错误')
        return self
```

Pydantic v2（2023 年发布）用 Rust（pydantic-core）重写了校验引擎，速度提升 5-50 倍，同时保持了完全兼容的 Python API。

> 📖 官方文档：[Pydantic Docs](https://docs.pydantic.dev/)

## 7.9 简化实现：MiniPydantic

下面是 53 行的简化版，展示 BaseModel 的核心原理——**读者可以运行、修改、加 print 看过程**。

> 完整可运行代码：[code/02_oop/mini_pydantic.py](https://github.com)

```python-run
from typing import get_type_hints, Any


class ValidationError(Exception):
    """校验错误"""
    pass


class MiniModel:
    """MiniPydantic：展示 BaseModel 核心机制"""
    __fields__: dict[str, type] = {}

    def __init_subclass__(cls, **kwargs: Any) -> None:
        """子类定义时自动收集类型注解"""
        super().__init_subclass__(**kwargs)
        hints = get_type_hints(cls)
        cls.__fields__ = {
            name: typ
            for name, typ in hints.items()
            if not name.startswith('_')
        }

    def __init__(self, **kwargs: Any) -> None:
        """自动校验 + 赋值 + 类型转换"""
        for field_name, field_type in self.__fields__.items():
            if field_name in kwargs:
                value = kwargs[field_name]
                if not isinstance(value, field_type):
                    try:
                        value = field_type(value)
                    except (ValueError, TypeError):
                        raise ValidationError(
                            f"字段 {field_name} 期望 {field_type.__name__}，"
                            f"收到了 {type(value).__name__}: {value}"
                        )
                setattr(self, field_name, value)
            else:
                raise ValidationError(f"缺少必填字段: {field_name}")

    def __repr__(self) -> str:
        fields = ', '.join(
            f"{name}={getattr(self, name)!r}"
            for name in self.__fields__
        )
        return f"{type(self).__name__}({fields})"


# ── 使用示例 ──
class User(MiniModel):
    name: str
    age: int

# 正常创建
u = User(name="Alice", age=30)
print(u)

# 类型转换：age 的 "25" → 25
u2 = User(name="Bob", age="25")
print(f"u2.age 的类型: {type(u2.age).__name__}")

# 校验失败
try:
    User(name="Eve", age="not a number")
except ValidationError as e:
    print(f"校验失败: {e}")
```

**三个关键技术点：**

| 机制 | 实现 | 对应 Python 特性 |
|------|------|-----------------|
| 自动收集字段 | `__init_subclass__` + `get_type_hints()` | 元类钩子 |
| 自动生成 `__init__` | 在 `__init__` 中遍历 `__fields__` 动态赋值 | `setattr` |
| 类型转换 + 校验 | `isinstance` 检查 → 失败时调类型构造函数 | 鸭子类型 |

## 7.10 类型标注 + Pydantic = 完整的类型故事

回顾这一章的旅程：

```
Python 类型标注（typing）           Pydantic
─────────────────────────           ────────────────────
标注，不约束                       标注，且约束
IDE / mypy 使用                     运行时使用
免费（不影响性能）                   有开销（但通常可忽略）
适合内部代码                       适合系统边界（API / DB / 配置）
```

**最佳实践：**

- 内部代码用 `typing` + mypy 做静态检查——免费
- 系统边界（API 请求、配置文件、数据库读写）用 Pydantic——花的性能换来安全
- 不要给每个内部函数返回的值都套一层 Pydantic Model

## 本章小结

- 类型标注不约束运行时行为（这是刻意的设计），价值在编码时：IDE 自动补全、mypy 静态检查、函数签名即文档、安全重构。
- `TypeVar` 实现基础泛型，`ParamSpec`（3.10+）捕获完整函数签名，`TypeVarTuple`（3.11+）处理变长类型参数——三者覆盖几乎所有泛型场景。
- `Protocol` 是鸭子类型的静态表达——只看结构不看继承链；`ABC` 要求显式继承或 `register()`，适合框架内部的行为契约。
- Pydantic 把类型标注当作运行时校验规则，自动进行类型强制转换和验证；v2 用 Rust（pydantic-core）重写引擎，速度提升 5-50 倍。
- 最佳实践：内部代码用 `typing` + mypy（零性能开销），系统边界（API/DB/配置）用 Pydantic（花少量性能换取数据安全）。
